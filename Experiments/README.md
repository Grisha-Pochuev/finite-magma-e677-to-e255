# Experiments

Эта папка хранит вычислительные прогоны проекта на GitHub Actions и других внешних машинах.

## Главные файлы веб-чата

- `START_HERE_WEB.md` — текущее состояние и постоянная память ChatGPT между чатами.
- `GITHUB_ACTIONS_RUNBOOK.md` — общий порядок подготовки, запуска и разбора GitHub Actions.

## Главное правило структуры

> Каждый новый прогон начинается, выполняется, разбирается и коммитится в одной папке с датой первого полного запуска.

Формат:

```text
Experiments/YYYY-MM-DD-short-name/
```

Дата — фактическая дата первого полного запуска. Если запуск до старта переносится на другой день, папку нужно переименовать заранее.

Повторная попытка после технического исправления остается в той же папке, если математическая задача и матрица по существу не изменились. Новая папка нужна только для нового эксперимента: другой гипотезы, существенно другой кодировки или другой матрицы.

Нельзя:

- готовить входы в одной папке, а результаты сохранять в другой;
- создавать отдельную папку только для retry того же эксперимента;
- коммитить итог вне папки, из которой запускалась матрица;
- смешивать результаты разных математических экспериментов в одной папке.

## Обязательное содержимое папки прогона

Минимальный набор:

- `README.md` — математическая цель, матрица и лимиты;
- все входные задачи;
- точная обертка запуска и контроль ресурсов;
- `workflow-used.yml` — точная копия использованного workflow;
- предварительный, затем итоговый `RESULTS.md`;
- номера всех run и соответствующие head SHA;
- `run-summary.csv` и/или `run-summary.json`;
- `CODEX_HANDOFF.md` — короткая справка для следующего аналитического шага;
- заметки о технических сбоях и различии между вычислительным и математическим итогом.

Полные журналы и частые ресурсные замеры можно оставить во временных GitHub-артефактах. В Git следует сохранять компактную проверенную сводку и данные, необходимые для воспроизведения.

## Жизненный цикл одного прогона

1. Создать одну датированную папку.
2. Подготовить пакет без полного запуска.
3. Провести статические проверки и короткий сквозной тест.
4. Получить прямое разрешение пользователя.
5. Запустить полный run и сразу записать его номер в ту же папку.
6. Разобрать результаты и создать компактную общую таблицу.
7. Обновить `README.md`, `RESULTS.md` и `CODEX_HANDOFF.md` в этой же папке.
8. Внести результат через одну временную ветку и слить одним squash-коммитом в `main`.
9. Обновить `START_HERE_WEB.md` и при новом общем уроке — runbook.

## Правила

1. Один математический прогон — одна датированная папка на всем пути от подготовки до итогового коммита.
2. Важный результат не должен оставаться только во временных артефактах GitHub Actions.
3. Исторические сценарии сохраняются как использованные, даже если позднее в них обнаружилась ошибка.
4. Полный запуск выполняется только после прямого решения пользователя.
5. Новый непроверенный сценарий не должен тем же коммитом сразу открывать длинную матрицу.
6. Перед большим прогоном обязателен короткий сквозной тест всех входов, профилей, журналов, классификации и загрузки артефактов.
7. Нельзя определять исход только по числовому коду или широкому поиску слова в журнале; учитываются финальный `SZS status`, монитор и обязательные файлы.
8. Технический зеленый статус не означает математического успеха, а тайм-аут не означает опровержения.
9. После завершения серии активный автоматический сценарий нужно удалить или обезвредить без случайного запуска новой матрицы.
10. Будущие workflow должны по возможности создавать один общий итоговый артефакт, чтобы веб-чату не приходилось разбирать каждый архив вручную.
