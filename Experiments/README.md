# Experiments

Эта папка хранит все вычислительные прогоны проекта на GitHub Actions и других внешних машинах.

## Главные файлы веб-чата

- `START_HERE_WEB.md` — текущее состояние проекта и постоянная память ChatGPT между чатами.
- `GITHUB_ACTIONS_RUNBOOK.md` — общая памятка по подготовке, проверке, запуску и разбору GitHub Actions. Перед каждым новым прогоном веб-чат обязан ее прочитать.

## Текущий прогон

- `2026-07-13-fixed-eta-zba-bridge/` — 20 заданий по 5 часов 50 минут для мостов `ZB*A=Ib`, `(ZB*Ib)*ZB=c`, `b*(ZB*A)=h`, прямой цели `A=z` и версии с условием плохой цели.

Предыдущая серия:

- `2026-07-11-fixed-eta/` — первый большой fixed-eta прогон и последующая диагностика машин, памяти и доказателей.

## Структура

- `YYYY-MM-DD-short-name/` — отдельная папка одного прогона.

В папке каждого прогона по возможности должны лежать:

- `README.md` — цель и устройство прогона;
- входные задачи и сценарии;
- `workflow-used.yml` — точная копия сценария GitHub Actions, который использовался;
- `RESULTS.md` — краткий проверенный итог;
- ссылки на запуск и артефакты;
- заметки о технических сбоях и о том, что нельзя считать математическим результатом.

## Правила

1. Один прогон — одна датированная папка.
2. Важный результат не должен оставаться только во временных артефактах GitHub Actions: краткий итог нужно сохранить в `RESULTS.md`.
3. Исторические сценарии сохраняются как использованные, даже если позднее в них обнаружилась ошибка.
4. Новые прогоны не должны автоматически запускаться от обычного изменения файлов. Полный запуск выполняется только после явного решения пользователя.
5. Перед большим прогоном обязателен короткий сквозной тест команд, всех входных файлов, всех разновидностей профилей, журналов, классификации результата и загрузки артефактов.
6. Первый коммит с новым непроверенным сценарием не должен одновременно запускать полную матрицу. Короткая проверка и разрешение длинного запуска — два отдельных этапа.
7. Нельзя оценивать исход только по числовому коду завершения доказателя. Нужно учитывать журнал, `SZS status`, причину контроля ресурсов и наличие обязательных файлов.
8. Все технические выводы отделяются от математических: сбой, тайм-аут или отсутствие доказательства не равны опровержению цели.
9. После завершения серии активный автоматический сценарий нужно удалить или обезвредить, чтобы поздняя правка не запустила новый полный прогон.
10. Если новый сбой дает общий урок, он добавляется в `GITHUB_ACTIONS_RUNBOOK.md`.
