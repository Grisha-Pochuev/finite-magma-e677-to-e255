# Памятка веб-чата по GitHub Actions

Дата последнего обновления: 13 июля 2026

Этот файл задает общий порядок подготовки, запуска и разбора вычислительных прогонов. Математическая цель каждого конкретного эксперимента хранится только в его датированной папке.

## Главные принципы

1. Длинный прогон не должен быть первым испытанием нового сценария.
2. Каждый прогон от подготовки до итогового коммита ведется в одной датированной папке.
3. Workflow должен заранее создавать компактный общий итог, чтобы после завершения не разбирать десятки архивов вручную.
4. Технический исход и математический результат всегда записываются отдельно.

## Одна папка на весь жизненный цикл

Для нового эксперимента создается:

```text
Experiments/YYYY-MM-DD-short-name/
```

Дата — дата первого полного запуска. В этой папке остаются подготовка, smoke-test, точный workflow, все run ID, повторные попытки, результаты и итоговый коммит.

Retry после технического исправления остается в той же папке, если математическая задача и матрица по существу не изменились. Новая папка нужна при новой гипотезе, существенно другой кодировке или другой матрице.

Обязательные файлы:

- `README.md`;
- входные задачи;
- обертка и контроль ресурсов;
- `workflow-used.yml`;
- предварительный, затем итоговый `RESULTS.md`;
- `run-summary.csv` и/или `run-summary.json`;
- `CODEX_HANDOFF.md`;
- ссылки на run и head SHA.

## Обязательная схема запуска

### Этап 1. Подготовить пакет без полного запуска

В датированной папке должны быть:

- математическая цель и точные подцели;
- все входные задачи;
- точная обертка запуска;
- контроль ресурсов;
- точная копия workflow;
- предварительный `RESULTS.md`;
- описание матрицы, лимитов и ожидаемых выходных файлов.

На этом этапе полный поиск не начинается.

### Этап 2. Статическая проверка

Проверить:

- синтаксис Bash и Python;
- существование каждого входного файла;
- число задач и строк матрицы;
- совпадение имен файлов в матрице;
- создание каталогов результатов до запуска;
- наличие и исполнимость доказателей;
- пути загрузки артефактов;
- отсутствие безусловного `exit 0`;
- наличие фиксированной схемы `summary.json` для каждого job;
- наличие финального сборщика результатов.

### Этап 3. Короткий сквозной тест

Короткая проверка использует ту же обертку и те же типы профилей, что и полный запуск.

Минимально необходимо:

- разобрать каждую входную задачу хотя бы одним доказателем;
- запустить каждый тип профиля;
- пройти через монитор дерева процессов;
- создать `solver.log`, коды завершения, `RESULT.txt` и `summary.json`;
- проверить загрузку артефакта;
- проверить работу финального сборщика;
- убедиться, что ожидаемый тайм-аут не становится технической ошибкой;
- убедиться, что неправильный параметр, отсутствующий файл и аварийное падение становятся красными.

Зеленый цвет без просмотра содержимого не считается достаточной проверкой.

### Этап 4. Отдельно разрешить полный запуск

Полный запуск выполняется только после прямого указания пользователя и успешного короткого теста.

Предпочтительно использовать:

```yaml
on:
  workflow_dispatch:
```

Допустим отдельный маркер или запускающий коммит, но первый коммит с новым непроверенным сценарием не должен сразу открывать длинную матрицу.

Нельзя полагаться только на точное название коммита: merge и исправления легко меняют его.

## Контракт результата одного matrix-job

Каждый job должен сохранять маленький `summary.json` с одинаковой схемой.

Минимальные поля:

```text
slot
problem
target
profile
raw_result
normalized_result
final_szs_status
raw_exit_code
monitor_exit_code
termination_reason
elapsed_seconds
peak_tree_rss_mib
min_mem_available_mib
max_swap_used_mib
cpu
commit_sha
```

Полезно также сохранять:

```text
artifact_name
theorem_found
model_found
technical_failure
worker_memory_limit_observed
```

`raw_result` хранит исходную подпись обертки. `normalized_result` хранит проверенный итог после учета финального статуса, монитора и журнала. Эти поля нельзя смешивать.

## Финальный job `collect`

После матрицы должен выполняться отдельный job:

```yaml
if: always()
needs: search
```

Он должен:

1. скачать маленькие summaries всех matrix-jobs;
2. отметить отсутствующие или поврежденные summaries;
3. создать `run-summary.csv`;
4. создать `run-summary.json`;
5. создать короткий `RUN_REPORT.md`;
6. посчитать доказательства, модели, тайм-ауты, memory guards и технические ошибки;
7. загрузить один общий итоговый артефакт.

Полные solver logs и частые ресурсные трассы остаются отдельными артефактами для углубленного расследования. Обычный итоговый разбор должен начинаться с общего артефакта.

## Как классифицировать завершение

Нельзя судить только по числовому коду и нельзя использовать широкое совпадение вроде `grep memory limit` как окончательный ответ.

Порядок проверки:

1. `SZS status Theorem` или `Unsatisfiable`;
2. `SZS status Satisfiable` или `CounterSatisfiable`;
3. явная ошибка синтаксиса, параметра, файла или аварийное падение;
4. причина остановки внешнего монитора;
5. финальный `SZS status` и финальная строка `Termination reason`;
6. только после этого сообщения отдельных портфельных работников о памяти или времени;
7. числовой код используется как дополнительный сигнал, а не как единственный источник.

Нормальные вычислительные исходы:

- доказательство или несовместность;
- модель или выполнимость;
- лимит времени;
- финальный внутренний предел памяти;
- защитная остановка по общей памяти;
- `GaveUp`, `Unknown`, `Open` или другой штатный неопределенный итог.

Технические ошибки:

- неправильный параметр;
- ошибка разбора входа;
- отсутствующий файл или программа;
- коды запуска `126` или `127`;
- незапланированное падение по сигналу;
- segmentation fault, assertion failure или fatal error;
- отсутствие обязательного журнала или summary.

Ожидаемый вычислительный исход может завершать job зеленым, но точная классификация должна быть записана. Техническая ошибка остается красной.

## Безопасность памяти

Для стандартного публичного Linux-runner исходить из 4 виртуальных ядер и примерно 15,6 ГиБ видимой памяти.

Проверенные исходные профили:

```text
Vampire: --mode casc --cores 4 -m 2500
Vampire: --mode casc --intent sat --cores 4 -m 2500
Vampire: --mode casc --cores 2 -m 5000
E prover: --auto-schedule=4 --memory-limit=2500
```

Контролировать:

- суммарный RSS всего дерева;
- `MemAvailable` машины;
- swap;
- модель CPU;
- точную команду;
- причину остановки.

Рабочая защита:

- мягкая остановка после трех последовательных измерений при RSS дерева около 12 ГиБ;
- либо при `MemAvailable` ниже 2,5 ГиБ;
- swap не считать дополнительной рабочей памятью.

## Правила workflow

- `fail-fast: false` для исследовательской матрицы;
- каталоги результатов создавать заранее;
- журналы и summaries загружать через `if: always()`;
- оставлять не менее 10 минут между полезным поиском и шестичасовым пределом job;
- сохранять точную копию workflow в папке прогона;
- добавлять финальный `collect`;
- не менять активный сценарий во время идущего run без необходимости;
- после завершения серии удалить или обезвредить автоматический запуск так, чтобы изменение не запустило новую матрицу.

## Что делать после запуска

В первые минуты проверить:

- прошла ли предварительная проверка;
- начали ли работать все ожидаемые jobs;
- нет ли мгновенных ошибок программы, параметров или входов;
- появляются ли summaries и артефакты;
- не растет ли память опасно быстро.

При красном крестике:

1. получить run ID, job ID и упавший шаг;
2. прочитать журнал именно этого шага;
3. не объявлять причину по значку или одному коду;
4. исправить минимальную подтвержденную причину;
5. снова провести короткий тест;
6. только затем повторять полный запуск.

Если журнал недоступен, прямо записать, что непосредственная причина не подтверждена. Можно исправить явную уязвимость, но нельзя выдавать гипотезу за найденную строку ошибки.

## Эффективный разбор завершенного run

1. Сразу записать run ID, head SHA и датированную папку.
2. Один раз получить список jobs.
3. Один раз получить список артефактов.
4. Сначала скачать общий итоговый артефакт `collect`.
5. Сырые артефакты отдельных jobs открывать только для доказательства, модели, аномалии или проверки сомнительной классификации.
6. Сверить числа общего отчета с количеством jobs и артефактов.
7. Отдельно посчитать доказательства, модели, тайм-ауты, solver memory limits, guards и технические ошибки.
8. Не считать отсутствие доказательства опровержением.
9. В Git сохранить компактную таблицу и отчет; полные повторяющиеся трассы не коммитить без специальной причины.

## Итоговый коммит

После разбора обновляются в той же датированной папке:

- `README.md`;
- `RESULTS.md`;
- `run-summary.csv` и/или `run-summary.json`;
- `CODEX_HANDOFF.md`;
- при необходимости исправленный анализатор или классификатор.

Затем:

1. создать одну временную ветку;
2. убедиться, что diff не содержит посторонних файлов;
3. создать PR;
4. слить squash-методом одним итоговым коммитом в `main`;
5. обновить `START_HERE_WEB.md` в том же PR либо сразу следующим ретроспективным коммитом.
