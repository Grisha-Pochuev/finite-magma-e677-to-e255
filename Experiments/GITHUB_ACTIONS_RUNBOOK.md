# Памятка веб-чата по GitHub Actions

Дата создания: 13 июля 2026

Этот файл предназначен прежде всего для ChatGPT в веб-чате. Он задает общий порядок подготовки, запуска и разбора вычислительных прогонов. Математическая цель каждого конкретного прогона хранится в его датированной папке.

## Главный принцип

Длинный прогон не должен одновременно быть первым испытанием нового сценария.

Нельзя в одном непроверенном коммите:

1. впервые добавить новую задачу, команды доказателей, обертку и сценарий;
2. сразу запустить 20 заданий почти на 6 часов.

Сначала должен пройти короткий сквозной тест. Полный запуск разрешается только отдельным действием после его успешного завершения.

## Обязательная схема запуска

### Этап 1. Подготовить пакет без полного запуска

В датированной папке должны быть:

- `README.md` с математической целью;
- все входные задачи;
- точная обертка запуска;
- контроль ресурсов;
- `workflow-used.yml`;
- пустой или предварительный `RESULTS.md`;
- точное описание матрицы заданий и лимитов.

На этом этапе полный поиск не должен начинаться.

### Этап 2. Статическая проверка

До обращения к GitHub Actions проверить:

- синтаксис всех сценариев оболочки;
- синтаксис Python;
- существование каждого входного файла;
- число задач и число строк матрицы;
- совпадение имен файлов в матрице с файлами на диске;
- создание каталога результатов до запуска программы;
- наличие и исполнимость доказателей;
- корректность пути загрузки артефактов;
- отсутствие безусловного `exit 0`.

### Этап 3. Короткий сквозной тест

Короткая проверка должна использовать не упрощенную похожую команду, а ту же обертку и те же профили, которые пойдут в полном запуске.

Минимально необходимо:

- разобрать каждую входную задачу хотя бы одним доказателем;
- запустить каждую разновидность профиля хотя бы один раз;
- пройти через контроль дерева процессов;
- создать `solver.log`, `RESULT.txt`, код завершения и сводку ресурсов;
- проверить загрузку артефакта;
- убедиться, что ожидаемый тайм-аут не становится красной технической ошибкой;
- убедиться, что неправильный параметр, отсутствующий файл и падение действительно становятся красной ошибкой.

Короткий тест считается успешным только после просмотра его состояния и журналов. Сам зеленый цвет без проверки содержимого недостаточен.

### Этап 4. Отдельно разрешить полный запуск

Полный запуск выполняется только после прямого указания пользователя и успешного короткого теста.

Предпочтительный способ:

- сценарий полного запуска имеет только `workflow_dispatch`;
- либо полный запуск включается отдельным файлом-маркером или отдельным запускающим коммитом;
- коммит, впервые добавляющий сценарий, не должен сам запускать длинную матрицу.

Если технически приходится запускать изменением файла сценария, сначала должен существовать отдельный уже проверенный короткий сценарий. Нельзя полагаться только на точное название коммита: такая защита хрупкая и легко ломается при исправлениях.

## Как классифицировать завершение доказателя

Нельзя судить только по числовому коду завершения. У разных доказателей нормальное отсутствие результата может иметь разные коды.

Нужно одновременно учитывать:

- текст `SZS status`;
- причину, записанную контролем ресурсов;
- сырой код завершения;
- наличие сигнала;
- сообщения о синтаксисе и параметрах;
- сообщения о памяти;
- наличие выходных файлов.

Нормальные вычислительные исходы:

- доказательство или несовместность;
- модель или выполнимость;
- лимит времени;
- внутренний предел памяти;
- защитная остановка по общей памяти;
- `GaveUp`, `Unknown`, `Open` или другой штатный неопределенный итог.

Технические ошибки:

- неправильный параметр;
- ошибка разбора входной задачи;
- отсутствующий файл или программа;
- код запуска `126` или `127`;
- падение по сигналу без запланированной остановки;
- segmentation fault, assertion failure или fatal error;
- отсутствие обязательных журналов и сводки.

Ожидаемые вычислительные исходы могут завершать задание зеленым цветом, но их точная классификация обязана быть записана в `RESULT.txt`. Техническая ошибка должна оставаться красной.

## Безопасность памяти

Для стандартной машины GitHub Actions исходить из 4 виртуальных ядер и примерно 15,6 ГиБ видимой памяти.

Проверенные исходные профили:

```text
Vampire: --mode casc --cores 4 -m 2500
Vampire: --mode casc --intent sat --cores 4 -m 2500
Vampire: --mode casc --cores 2 -m 5000
E prover: --auto-schedule=4 --memory-limit=2500
```

Обязательный контроль:

- суммарная память всего дерева процессов;
- `MemAvailable` всей машины;
- использование подкачки;
- модель процессора;
- точная команда;
- причина остановки.

Рабочая защитная граница:

- мягко остановить после трех последовательных измерений при RSS дерева около 12 ГиБ;
- либо при `MemAvailable` ниже 2,5 ГиБ;
- не считать подкачку дополнительной бесплатной оперативной памятью.

## Правила сценария GitHub Actions

- `fail-fast: false` для исследовательской матрицы;
- журналы и сводки загружать через `if: always()`;
- создавать каталог результата заранее;
- оставлять не менее 10 минут между полезным лимитом поиска и шестичасовым пределом задания;
- сохранять точную копию использованного сценария в папке прогона;
- после завершения серии удалить или обезвредить активный автоматический сценарий, чтобы обычные правки не запустили новый полный прогон;
- не менять активный сценарий во время идущего прогона без необходимости.

## Что делать после запуска

В первые минуты проверить:

- прошла ли предварительная проверка;
- начали ли выполняться все ожидаемые задания;
- нет ли мгновенных ошибок загрузки программы, параметров или входных файлов;
- создаются ли артефакты;
- не растет ли память опасно быстро.

При красном крестике:

1. сначала получить конкретный номер запуска, задания и шаг;
2. прочитать журнал упавшего шага;
3. не объявлять причину по одному коду или догадке;
4. исправить минимальную подтвержденную причину;
5. снова провести короткий тест;
6. только затем перезапускать полный поиск.

Если журналы недоступны через подключение, нужно прямо сказать, что причина не подтверждена. Допустимо исправить явную уязвимость сценария, но нельзя выдавать предположение за найденную строку ошибки.

## Разбор результатов

После завершения необходимо отдельно посчитать:

- доказательства;
- модели;
- тайм-ауты;
- пределы памяти;
- защитные остановки;
- технические ошибки;
- задания без решающего результата.

Отсутствие доказательства за 5 часов 50 минут не является опровержением математического утверждения.

Итог записывается в:

1. `RESULTS.md` папки прогона;
2. `Experiments/START_HERE_WEB.md`;
3. при необходимости `Experiments/README.md`.

## Короткий контрольный список перед полным запуском

- [ ] Пользователь прямо разрешил полный запуск.
- [ ] Математическая постановка сверена со свежим пакетом.
- [ ] Все входные задачи разобраны доказателем.
- [ ] Все профили прошли короткий тест через настоящую обертку.
- [ ] Проверены нормальные и аварийные исходы.
- [ ] Нет безусловного `exit 0`.
- [ ] Память ограничена на одного работника и контролируется для всего дерева.
- [ ] Каталоги результатов создаются заранее.
- [ ] Артефакты загружаются при любом исходе.
- [ ] Полезный лимит меньше лимита GitHub минимум на 10 минут.
- [ ] Полный запуск отделен от первого добавления сценария.
- [ ] После запуска известны ссылка и номер прогона.

## Урок запуска 13 июля 2026

При подготовке ZBA-прогона короткая проверка была встроена в тот же сценарий, который должен был сразу открыть двадцать долгих заданий. Кроме того, проверка считала нормальными только заранее перечисленные коды `0`, `1` и `124`. Это было слишком хрупко: штатное завершение доказателя могло стать красным техническим сбоем.

После ошибки проверка была расширена: все задачи разбираются, два профиля проходят через настоящую обертку, а классификация учитывает журналы и тип результата. Однако более правильное глобальное решение состоит в разделении короткой проверки и разрешения полного запуска на два отдельных этапа.
