# START HERE — WEB CHAT

Дата последнего обновления: 12 июля 2026

## Для кого этот файл

Это постоянная память для ChatGPT в веб-чате, когда пользователь закрывает один чат и продолжает работу в другом.

Этот файл не является рабочей инструкцией для Codex. Codex не должен выбирать по нему математический фронт или менять свои основные файлы состояния, если пользователь прямо этого не попросил.

## Как начинать новый веб-чат

1. Прочитать этот файл целиком.
2. Открыть `Experiments/README.md`.
3. Открыть `README.md` и `RESULTS.md` в самой свежей датированной папке прогона.
4. Не запускать новый прогон, пока пользователь явно не попросит это сделать.

## Цель проекта

Исследуется открытая конечная импликация:

`E677 -> E255`

Нужно доказать, что всякая конечная магма, удовлетворяющая E677, удовлетворяет E255, либо найти конечный контрпример.

Основная математическая работа ведется Codex и хранится в обычных файлах проекта. Веб-чат помогает:

- разбирать результаты GitHub Actions;
- планировать вычислительные прогоны;
- проверять техническую корректность сценариев;
- сохранять итоги прогонов;
- объяснять пользователю происходящее простыми словами.

## Текущее математическое состояние

На 11 июля 2026 активна фиксированная ветка `eta = r` внутри строгого period-3 остатка.

Непосредственная цель:

`r*A = U`

Эквивалентная целевая форма для автоматического доказателя:

`A = z`

Подготовленная постановка использует E677, левое сокращение и известные якорные равенства фиксированной ветки.

Важно: автоматический прогон 11–12 июля не нашел доказательства и частично оказался технически некорректным. Он не закрыл математическую цель.

## Последний прогон

Папка:

`Experiments/2026-07-11-fixed-eta/`

GitHub Actions:

`https://github.com/Grisha-Pochuev/finite-magma-e677-to-e255/actions/runs/29168947710`

Изначально планировались 20 заданий:

- пять вариантов постановки;
- до четырех ядер на задание;
- примерно 5 часов 40 минут поиска.

Проверенный итог:

- 8 обычных запусков E prover дошли до лимита времени и не нашли доказательство;
- 2 запуска E с `ground-boost` завершились технической ошибкой;
- 5 запусков Vampire в режиме `casc` были прерваны сигналом завершения с кодом 143 примерно через 4–5 минут;
- 5 запусков Vampire с записью `casc_sat` не являлись полноценным поиском, потому что текущая версия Vampire не принимает `casc_sat` как значение `--mode`;
- зеленый статус части ошибочных заданий был ложным следствием безусловного `exit 0` в сценарии.

Полный разбор сохранен в `Experiments/2026-07-11-fixed-eta/RESULTS.md`.

## Главные технические уроки

1. Перед большим запуском нужен отдельный сквозной тест E и Vampire на 60–120 секунд.
2. Нельзя завершать сценарий безусловным `exit 0` после любой ошибки.
3. Для современного Vampire SAT-направление должно задаваться через `--mode casc --intent sat`, а не через `--mode casc_sat`.
4. Нужно контролировать общий расход памяти всего дерева процессов, а не только задавать высокий предел одному процессу.
5. Нельзя отдавать доказателю почти все 15 ГиБ памяти машины; нужен запас для системы и служебных процессов.
6. Следует измерять модель процессора, общий расход памяти, подкачку и скорость обработки состояний.
7. Большой прогон запускается только вручную после проверки короткого прогона.
8. Результаты и технические сбои нужно коммитить в датированную папку, а не оставлять только во временных артефактах GitHub.

## Следующий вычислительный шаг

Ничего не запускать автоматически.

Перед следующим большим математическим прогоном сначала подготовить отдельный диагностический и проверочный пакет:

- паспорт машины GitHub: Intel или AMD, модель процессора, память, кэши;
- измерение реальной пропускной способности памяти;
- короткая проверка E и Vampire;
- контроль общей памяти дерева процессов;
- корректная классификация кодов завершения;
- обязательная загрузка журналов даже после аварии.

После этого подобрать безопасные настройки ядер и памяти и только затем обсуждать новый полный прогон.

## Правило передачи между чатами

1. этот файл;

3. при необходимости `Experiments/README.md`.
