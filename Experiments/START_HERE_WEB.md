# START HERE — WEB CHAT

Дата последнего обновления: 13 июля 2026

## Для кого этот файл

Это постоянная память для ChatGPT в веб-чате, когда пользователь закрывает один чат и продолжает работу в другом.

Этот файл не является рабочей инструкцией для Codex. Codex не должен выбирать по нему математический фронт или менять свои основные файлы состояния, если пользователь прямо этого не попросил.

## Как начинать новый веб-чат

1. Прочитать этот файл целиком.
2. Открыть `Experiments/README.md`.
3. Перед подготовкой или разбором GitHub Actions прочитать `Experiments/GITHUB_ACTIONS_RUNBOOK.md`.
4. Открыть `README.md` и `RESULTS.md` в самой свежей датированной папке прогона.
5. Не запускать новый прогон, пока пользователь явно не попросит это сделать.

## Цель проекта

Исследуется открытая конечная импликация:

```text
E677 -> E255
```

Нужно доказать, что всякая конечная магма, удовлетворяющая E677, удовлетворяет E255, либо найти конечный контрпример.

Основная математическая работа ведется Codex и хранится в обычных файлах проекта. Веб-чат помогает:

- разбирать результаты GitHub Actions;
- планировать вычислительные прогоны;
- проверять техническую корректность сценариев;
- сохранять итоги прогонов;
- объяснять пользователю происходящее простыми словами.

## Обязательный порядок новых запусков

Главное правило:

> Длинный прогон не должен быть первым испытанием нового сценария.

Перед полным запуском обязательны отдельные стадии:

1. подготовить папку, задачи, команды и сценарий без запуска длинной матрицы;
2. провести статическую проверку;
3. провести короткий сквозной тест всех входных задач и всех разновидностей профилей через настоящую обертку;
4. посмотреть состояние и журналы короткого теста;
5. только отдельным действием разрешить полный запуск.

Нельзя впервые добавлять новый сценарий и тем же непроверенным коммитом сразу открывать 20 заданий почти на 6 часов.

Подробные правила, классификация кодов завершения и контрольный список находятся в:

```text
Experiments/GITHUB_ACTIONS_RUNBOOK.md
```

## Текущее математическое состояние

На 12 июля 2026 активна фиксированная ветка `eta = r` внутри строгого period-3 остатка.

Свежий пакет пользователя сузил прежнюю цель `r*A=U` до более короткого моста:

```text
ZB*A=Ib  =>  A=z.
```

Эквивалентная более мягкая форма:

```text
b*(ZB*A)=h.
```

Текущая узкая промежуточная цель:

```text
(ZB*Ib)*ZB=c.
```

Если она доказана, E677 на строке `ZB` и входе `Ib` дает `ZB*A=Ib`.

Все 3258 сохраненных фиксированных eta-return строк удовлетворяют `ZB*z=Ib`, `ZB*A=Ib` и `b*(ZB*A)=h`. Это вычислительное свидетельство, но не доказательство.

Папка:

```text
Experiments/2026-07-13-fixed-eta-zba-bridge/
```

Первый запускающий коммит:

```text
a130e97addde2060ea1596f757f180c0d20cdc92
```

После красных технических состояний была обнаружена хрупкая часть сценария: предварительная проверка считала нормальными только заранее перечисленные коды `0`, `1` и `124`. Точная строка ошибки из журналов через подключение веб-чата не была доступна, поэтому это нельзя называть полностью подтвержденной единственной причиной. Однако сама логика была объективно небезопасной и исправлена.

Исправление и повторный запуск:

```text
PR: https://github.com/Grisha-Pochuev/finite-magma-e677-to-e255/pull/9
commit: 9fd857118fbf6a7aafc73da53c3092b01a17c6d4
```

Исправленная короткая проверка:

- разбирает все пять входных задач;
- запускает E prover и Vampire через настоящую обертку и контроль памяти;
- различает штатный неопределенный исход и техническое падение;
- не считает `Timeout`, `GaveUp`, `ResourceOut` и похожие штатные исходы аварией;
- оставляет ошибки синтаксиса, параметров, файлов и сигнальные падения красными.

Страница сценария:

```text
https://github.com/Grisha-Pochuev/finite-magma-e677-to-e255/actions/workflows/fixed-eta-zba-bridge-2026-07-13.yml
```

В полном прогоне 20 независимых заданий. Каждое получает ровно 350 минут, то есть 5 часов 50 минут полезного поиска.

Проверяются пять форм:

1. `ZB*A=Ib`;
2. `(ZB*Ib)*ZB=c`;
3. `b*(ZB*A)=h`;
4. контрольная прямая цель `A=z`;
5. `ZB*A=Ib` с условием плохой цели `forall x, x*b != b`.

Каждая форма проверяется четырьмя профилями:

- E prover: 4 работника по 2500 МБ;
- Vampire CASC: 4 работника по 2500 МБ;
- Vampire CASC: 2 работника по 5000 МБ;
- Vampire CASC с `--intent sat`: 4 работника по 2500 МБ.

## Проверенные безопасные настройки

Основной Vampire:

```text
--mode casc --cores 4 -m 2500
```

Vampire в SAT-направлении:

```text
--mode casc --intent sat --cores 4 -m 2500
```

Дополнительный профиль:

```text
--mode casc --cores 2 -m 5000
```

E prover:

```text
--auto-schedule=4 --memory-limit=2500
```

Обязательные меры:

- измерять память всего дерева процессов;
- мягко останавливать после трех опасных измерений при RSS дерева около 12 ГиБ или `MemAvailable` ниже 2,5 ГиБ;
- сохранять модель процессора, точную команду, журнал, таблицу памяти и классификацию результата;
- не маскировать тайм-аут под доказательство или опровержение;
- оставлять технический сбой красной ошибкой;
- загружать артефакты через `if: always()`.

## Следующий шаг веб-чата

Пока повторный ZBA-прогон идет, ничего нового не запускать.

После его завершения:

1. получить номер запуска и все 20 артефактов;
2. отделить доказательство, модель, тайм-аут, предел памяти и техническую ошибку;
3. проверить доказательный объект при наличии `SZS status Theorem`;
4. записать итог в `Experiments/2026-07-13-fixed-eta-zba-bridge/RESULTS.md`;
5. обновить этот файл;
6. обезвредить или удалить активный автоматический сценарий, чтобы поздняя правка не запустила новый полный прогон.

## Правило передачи между чатами

После каждого существенного прогона веб-чат должен обновить:

1. этот файл;
2. `RESULTS.md` соответствующей датированной папки;
3. при необходимости `Experiments/README.md`;
4. `Experiments/GITHUB_ACTIONS_RUNBOOK.md`, если обнаружен новый общий технический урок.

В новом чате пользователю достаточно написать:
