## Для кого этот файл

Это постоянная память для ChatGPT в веб-чате, когда пользователь закрывает один чат и продолжает работу в другом.

Этот файл не является рабочей инструкцией для Codex. Codex не должен выбирать по нему математический фронт или менять свои основные файлы состояния, если пользователь прямо этого не попросил.

## Как начинать новый веб-чат

1. Прочитать этот файл целиком.
2. Открыть `Experiments/README.md`.
3. Открыть `README.md` и `RESULTS.md` в самой свежей датированной папке прогона.
4. Не запускать новый прогон, пока пользователь явно не попросит это сделать.

## Цель проекта

Исследуется открытая конечная импликация:

`E677 -> E255`

Нужно доказать, что всякая конечная магма, удовлетворяющая E677, удовлетворяет E255, либо найти конечный контрпример.

Основная математическая работа ведется Codex и хранится в обычных файлах проекта. Веб-чат помогает:

- разбирать результаты GitHub Actions;
- планировать вычислительные прогоны;
- проверять техническую корректность сценариев;
- сохранять итоги прогонов;
- объяснять пользователю происходящее простыми словами.

## Текущее математическое состояние

На 11 июля 2026 активна фиксированная ветка `eta = r` внутри строгого period-3 остатка.

Непосредственная цель:

`r*A = U`

Эквивалентная целевая форма для автоматического доказателя:

`A = z`

Подготовленная постановка использует E677, левое сокращение и известные якорные равенства фиксированной ветки.

Большой автоматический прогон 11–12 июля не нашел доказательства и частично оказался технически некорректным. Диагностический прогон 12 июля был только техническим и тоже не меняет математический статус.

## Диагностический прогон 12 июля

GitHub Actions:

`https://github.com/Grisha-Pochuev/finite-magma-e677-to-e255/actions/runs/29207844306`

Run ID: `29207844306`

Все 20 коротких измерительных заданий, предварительная проверка и итоговая сборка завершились успешно.

Главные результаты:

1. В Vampire предел `-m` применяется к отдельному процессу. Режим `--cores 4` создает четыре работника. Поэтому старая настройка `--cores 4 -m 14500` теоретически разрешала работникам около `58 ГБ` суммарно на машине с `15,6 ГиБ` физической памяти. Это наиболее вероятная причина старого внешнего завершения, хотя код 143 сам по себе не доказывает OOM.
2. Все безопасные профили дошли до 90-секундного лимита без подкачки и без событий `oom` или `oom_kill`.
3. В одном и том же запуске GitHub выдал разные процессоры: 14 × AMD EPYC 7763, 4 × AMD EPYC 9V74, 1 × Intel Xeon 8573C и 1 × Intel Xeon 8370C.
4. Выбрать Intel или AMD для обычного `ubuntu-24.04` нельзя. Нужно опираться на общие характеристики машины и записывать фактически выданную модель.
5. Физическая память во всех заданиях была около `15,6 ГиБ`; наблюдалась подкачка `3 ГиБ`, но в безопасном тесте она не использовалась.

Полный разбор:

`Experiments/2026-07-11-fixed-eta/runner-diagnostics/RESULTS.md`

## Проверенные безопасные настройки

Основной Vampire:

```text
--mode casc --cores 4 -m 2500
```

Vampire в SAT-направлении:

```text
--mode casc --intent sat --cores 4 -m 2500
```

E prover:

```text
--auto-schedule=4 --memory-limit=2500
```

Дополнительный профиль для отдельных тяжелых по памяти стратегий:

```text
--mode casc --cores 2 -m 5000
```

Основной смысл: суммарный теоретический бюджет работников держится около `10 ГБ`, а системе остается несколько гигабайт. Нельзя стремиться занять все 16 ГБ: подкачка способна замедлить поиск сильнее, чем дополнительная память поможет.

## Обязательные правила будущего полного запуска

1. Большой прогон запускается только после прямого указания пользователя.
2. Перед ним сохраняется короткая предварительная проверка команд.
3. Нет безусловного `exit 0`.
4. Журналы загружаются через `if: always()`.
5. Измеряется память всего дерева процессов.
6. Различаются тайм-аут, предел памяти, код 137, код 143, ошибка параметров и математический результат.
7. Желательно мягко останавливать задание, если дерево приближается примерно к `12 ГиБ` или `MemAvailable` несколько раз подряд падает ниже `2,5 ГиБ`.
8. В каждом задании сохраняются модель процессора и короткое измерение скорости памяти.
9. Для Vampire используется смесь широких заданий `4 × 2500 МБ` и небольшого числа глубоких заданий `2 × 5000 МБ`.
