# START HERE — WEB CHAT

Дата последнего обновления: 13 июля 2026

## Для кого этот файл

Это постоянная память ChatGPT в веб-чате между разными чатами.

Этот файл не заменяет математические файлы Codex. Он хранит только состояние веб-чата, правила GitHub Actions и ссылки на последний прогон.

## Как начинать новый веб-чат

1. Прочитать этот файл один раз целиком.
2. Открыть `Experiments/README.md`.
3. Если работа связана с GitHub Actions, прочитать `Experiments/GITHUB_ACTIONS_RUNBOOK.md`.
4. Открыть `README.md`, `RESULTS.md` и компактную итоговую таблицу в самой свежей датированной папке.
5. Зафиксировать четыре вещи: текущую папку эксперимента, номер run, последний итоговый коммит и разрешил ли пользователь новый запуск.
6. Не запускать новый прогон без прямого указания пользователя.

Не нужно повторно перечитывать большие исторические файлы, если вопрос относится только к уже указанному run и его папке.

## Цель проекта

Исследуется конечная импликация:

```text
E677 -> E255
```

Нужно либо доказать, что всякая конечная магма с E677 удовлетворяет E255, либо найти конечный контрпример.

Основную математику ведет Codex. Веб-чат отвечает прежде всего за:

- подготовку и проверку внешних вычислительных прогонов;
- запуск только после разрешения пользователя;
- разбор GitHub Actions;
- сохранение результатов и технической истории;
- простое объяснение результата пользователю;
- передачу компактной справки Codex.

## Где хранится истина

Чтобы не дублировать и не рассинхронизировать сведения:

- математический фронт хранится в основных файлах проекта Codex;
- общие технические правила GitHub Actions хранятся в `Experiments/GITHUB_ACTIONS_RUNBOOK.md`;
- полный жизненный цикл конкретного прогона хранится в его датированной папке;
- этот файл хранит только последнюю сводку и следующий шаг веб-чата.

## Главное правило: один прогон — одна датированная папка

Каждый новый прогон с момента подготовки и до итогового коммита ведется в одной папке:

```text
Experiments/YYYY-MM-DD-short-name/
```

Дата в имени — дата первого полного запуска этого эксперимента. Если запуск перенесен на другой день до старта, папку нужно переименовать до запуска.

В одной и той же папке должны оставаться:

- постановка задачи;
- входные файлы;
- обертки и контроль ресурсов;
- точная копия использованного workflow;
- smoke-test и его выводы;
- все номера запусков и повторных попыток;
- компактная таблица результатов;
- итоговый `RESULTS.md`;
- короткая справка для Codex;
- финальный коммит результата.

Повторный запуск после технического исправления остается в той же папке, если математическая задача и матрица по существу те же. Новая папка создается только для нового эксперимента: другой гипотезы, существенно другой кодировки или другой матрицы.

Нельзя готовить прогон в одной папке, сохранять результаты в другой и коммитить итог в третьем месте.

## Оптимизированный порядок работы веб-чата

### До запуска

1. Создать одну датированную папку.
2. Заполнить `README.md` и предварительный `RESULTS.md`.
3. Сохранить `workflow-used.yml`.
4. Проверить все входы, профили и аварийные исходы коротким сквозным тестом.
5. Только после прямого разрешения пользователя открыть полный запуск.

### Во время запуска

1. Сразу сохранить номер run и ссылку в `RESULTS.md`.
2. Проверить smoke-test и начало всех jobs.
3. Не менять математическую постановку в середине идущего прогона.
4. При красном состоянии сначала читать конкретный job и шаг, а не угадывать по значку.

### После запуска

1. Один раз получить список jobs и список артефактов.
2. Использовать общий итоговый артефакт, если workflow его создал.
3. Отдельно считать доказательства, модели, тайм-ауты, защитные остановки и технические ошибки.
4. Сохранять исходную подпись программы и отдельный проверенный нормализованный итог.
5. Коммитить компактную сводку, а не многомегабайтные повторяющиеся замеры каждые две секунды.
6. Обновить `README.md`, `RESULTS.md`, итоговую CSV/JSON и справку Codex в той же папке.
7. Внести изменения через одну временную ветку и слить одним squash-коммитом в `main`.
8. Обновить этот файл и при необходимости общий runbook.

## Требование к будущим workflow

Чтобы следующий разбор не требовал скачивать двадцать архивов по одному:

- каждый matrix-job должен создавать маленький `summary.json` с фиксированными полями;
- финальный job `collect` с `if: always()` должен скачать все summaries;
- `collect` должен создать единые `run-summary.csv`, `run-summary.json` и краткий Markdown-отчет;
- единый итоговый артефакт должен загружаться даже при частичных сбоях;
- полные журналы и частые замеры ресурсов остаются временными артефактами, а в Git попадает компактная сводка.

Минимальные поля одного job:

```text
slot
problem
target
profile
raw_result
normalized_result
final_szs_status
exit_code
termination_reason
elapsed_seconds
peak_tree_rss_mib
min_mem_available_mib
max_swap_used_mib
cpu
artifact_id
artifact_sha256
```

## Правило классификации

Нельзя определять результат широким поиском слов вроде `memory limit` по всему журналу.

Приоритет проверки:

1. доказательство или несовместность;
2. модель или выполнимость;
3. явная техническая ошибка;
4. причина остановки внешнего монитора;
5. финальный `SZS status` и финальная причина завершения доказателя;
6. только затем вспомогательные сообщения отдельных работников о времени или памяти.

Исходная машинная подпись сохраняется отдельно от проверенного нормализованного итога.

## Текущее математическое состояние

Активная область — фиксированная ветка `eta = r` внутри строгого period-3 остатка.

Проверялись мосты:

```text
ZB*A=Ib
(ZB*Ib)*ZB=c
b*(ZB*A)=h
A=z
```

а также `ZB*A=Ib` с дополнительным условием плохой цели.

Вычислительные данные поддерживают эти равенства на сохраненных фиксированных eta-return строках, но общего доказательства пока нет.

## Ретроспектива завершенной цепочки

Что сработало хорошо:

- один эксперимент действительно прошел путь от подготовки до итогового коммита в одной папке;
- удалось получить и проверить все 20 артефактов;
- математический исход был отделен от технического зеленого статуса;
- память контролировалась безопасно;
- результат сохранен в читаемой и машинно-читаемой форме.

Что нужно упростить в следующий раз:

- не скачивать каждый артефакт вручную, а заранее делать финальный `collect`;
- не дублировать подробные технические правила в нескольких файлах;
- не хранить в Git полные повторяющиеся ресурсные трассы;
- сразу записывать run ID, head SHA и папку эксперимента;
- проверять финальный статус доказателя раньше широких текстовых совпадений;
- готовить справку Codex как штатный файл папки, а не писать ее заново после коммита;
- собирать все изменения результата в одну ветку и один squash-коммит.
