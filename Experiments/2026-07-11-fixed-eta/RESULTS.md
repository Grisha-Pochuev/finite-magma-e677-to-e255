## Vampire

Из десяти заданий Vampire:

### Режим `casc`

Пять заданий `casc` были прерваны примерно через 4–5 минут.

В доступных журналах присутствуют:

```text
The runner has received a shutdown signal
Process completed with exit code 143
```

Это означает внешнее завершение процесса сигналом `TERM`. По этим строкам нельзя строго доказать, что причиной была именно нехватка памяти.

Однако конфигурация была опасной:

```text
--cores 4 -m 14500
```

Диагностика 12 июля уточнила, что `-m` применяется к отдельному процессу Vampire, а портфель создает несколько дочерних процессов. Поэтому четыре работника могли иметь суммарный теоретический предел около `58 ГБ` на машине с примерно `15,6 ГиБ` физической памяти.

### Запись `casc_sat`

Пять заданий с `casc_sat` не выполнили задуманный SAT-поиск.

В использованной версии Vampire `casc_sat` не является допустимым значением параметра `--mode`. Корректная форма для SAT-направления должна использовать:

```text
--mode casc --intent sat
```

Ошибочные задания могли отображаться зелеными, потому что `run_one.sh` в конце всегда выполнял:

```text
exit 0
```

Таким образом, зеленый статус этих заданий не является подтверждением нормальной работы.

## Что прогон действительно сообщил

1. Восемь длительных стратегий E prover не нашли доказательство в заданное время.
2. Текущая постановка достаточно сложна для обычного автоматического поиска.
3. Конфигурация Vampire и обработка кодов завершения были недостаточно проверены.
4. Полный портфель Vampire придется провести заново только после отдельного короткого сквозного теста.
5. Нельзя делать математические выводы из красного или зеленого значка без чтения журнала.

## Что исправить перед будущим прогоном

- выполнить 60–120-секундный сквозной тест всех команд;
- использовать `--mode casc --intent sat`;
- убрать безусловный `exit 0`;
- различать тайм-аут, отсутствие доказательства, ошибку параметров, код 137, код 143 и найденное доказательство;
- загружать журналы через `if: always()` даже после сбоя;
- измерять общий расход памяти всего дерева процессов;
- оставить несколько гигабайт запаса для системы;
- сначала провести отдельный диагностический прогон характеристик машин GitHub;
- большой прогон запускать только вручную после проверки.
