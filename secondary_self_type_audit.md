План из этого аудита выполнен. Результат оформлен в:

```text
secondary_self_type_relay_lemma.md
```

Итог:

```text
u=2, q=2 -> closed
```

Этот файл остается как запись аудита ошибки `3*0=h` и вывода валидного
релейного слоя.

## Зачем создан этот файл

Последний этап начал закапываться в конкретный хвост `u=2,q=2`. При аудите
обнаружена слабая связка: в рабочую цепочку была добавлена формула `3*0=h`,
которая не следует из уже доказанных ребер. Этот файл фиксирует чистую,
валидную форму дальнейшего движения.

## Текущий открытый узел

```text
case 45
7*0=4
6*6=8
6*5=7
u=7*6=6*0=2
q=2*6=2
```

Закрыто внутри `u=2`:

```text
q=0
q=6
```

Открыт вторичный self-type:

```text
2*6=2
```

То есть активная строка должна перейти со строки `6` на строку `2`.

## Что было неверно

Была записана цепочка:

```text
s=6*2
h=s*6
2*h=0
3*0=h
```

Первые три строки корректны. Последняя строка `3*0=h` не доказана.

Проверка:

из ребра `6*0=2` обратная цепочка дает:

```text
0 = 2*((6*2)*6)
```

Если

```text
s=6*2
h=s*6
```

то действительно:

```text
2*h=0
```

Но из `0*2=3` не следует `3*0=h`. Для релейного вывода такого вида
понадобилось бы другое ребро строки `2`, например `2*h=1`, а не `2*h=0`.

Дополнительная проверка после аудита:

```text
s=6
```

закрывается и без условия `3*0=h`:

```text
status: none
time: около 12 секунд
```

Значит ошибка была в объяснении, а не в самом факте закрытия `s=6`.

## Валидная широкая форма

Ролевая версия для вторичного self-type:

```text
b3*0=u
u*b3=u
```

Тогда из ребра `b3*0=u` следует:

```text
s=b3*u
h=s*b3
u*h=0
```

В текущих числах:

```text
b3=6
u=2
s=6*2
h=s*6
2*h=0
```

Это настоящий релейный выход из вторичного self-type.

## Следующий валидный релейный уровень

Теперь надо использовать новое ребро:

```text
u*h=0
```

Его обратная цепочка дает:

```text
r=u*0
g=r*u
0*g=h
```

В текущих числах:

```text
r=2*0
g=r*2
0*g=h
```

Так как строка `0` уже является полным 9-циклом, условие `0*g=h` не является
случайной новой клеткой: оно связывает `h` с известной циклической шкалой.

## Кандидат-лемма

Если в ветке полного 9-цикла появляется вторичный self-type

```text
b3*0=u
u*b3=u
```

то продолжение должно идти не по старому хвосту строки `b3`, а по двухшаговой
схеме:

```text
1. b3*0=u
   s=b3*u
   h=s*b3
   u*h=0

2. u*h=0
   r=u*0
   g=r*u
   0*g=h
```

Это превращает локальный хвост в релейную схему между строками `b3`, `u` и
известным циклом строки `0`.

## Короткая диагностика после аудита

Проведена короткая локальная диагностика без недоказанной связи `3*0=h`.

Для базового узла `u=2,q=2`:

```text
row 6 domain: 22
row 2 domain: 1244
6*2 in {1,3,4,6}
```

Это означает: на самом входе строка `6` еще компактнее строки `2`, но значение
`q=u` уже указывает, что следующий настоящий механизм должен объяснить переход
к строке `2`.

Отдельно проверено:

```text
6*2=6 -> closed without 3*0=h
```

Остаются:

```text
6*2 in {1,3,4}
```

Но их не надо продолжать как простой список случаев до проверки новой
двухуровневой схемы.

Не продолжать старое дробление:

```text
s=6*2 in {1,3,4}
```

Значение `s=6` уже перепроверено и закрыто без ошибочной связи. Для оставшихся
`s=1,3,4` сначала проверить новую валидную схему:

```text
s=6*2
h=s*6
2*h=0
r=2*0
g=r*2
0*g=h
```

Цель проверки не закрыть еще один хвост, а понять, сжимает ли этот слой строку
`2` так же, как `u-transfer` сжимал строку `6`.

## Практический следующий вопрос

Короткая диагностика должна ответить:

```text
после добавления r=2*0, g=r*2, 0*g=h
первая активная строка остается 2
или реле уходит в новую строку r?
```

Если активная строка остается `2` и домен резко сжимается, это кандидат на
настоящую лемму вторичного self-type. Если нет, нужно сменить разрез, а не
повышать лимиты.
