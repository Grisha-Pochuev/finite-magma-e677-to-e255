# Agent Workflow

Дата: 2026-05-23.

Этот файл описывает, как агент должен продолжать исследование дальше. Он нужен,
чтобы работа не превращалась в длинный список закрытых случаев.

## Главный принцип

Каждое вычисление должно быстро превращаться в структурную фразу.

## Область обычного поиска

Обычный поиск по проекту не должен читать исторические архивы и копии,
подготовленные для публикации. Они исключены через `.rgignore`:

```text
archive/
github_publish/
github_upload_update_2026-06-08/
```

Искать в этих каталогах только при проверке конкретной старой записи или
публикационного пакета, явно задавая нужный путь. Не использовать режимы
`rg -u` или `rg -uu` для обычного исследования.

Не цель:

```text
закрыть еще одну ветку
```

Цель:

```text
понять, почему такой тип веток закрывается
```

## Правило аудита вывода

Перед продолжением трудного хвоста агент обязан отделить доказанные следствия
от диагностических добавок.

Если в рабочей цепочке появляется новая формула вида:

```text
новая клетка = старое выражение
```

нужно явно указать, из какого ребра E677 она следует. Если такого ребра нет,
формулу нельзя использовать как условие закрытия. Ее можно сохранить только как
эвристическую пробу, пометив это в журнале.

Это правило добавлено после аудита 2026-05-24: связь `3*0=h` в хвосте
`u=2,q=2` не была доказана и не должна использоваться как факт.

## Рабочий цикл

1. Прочитать `CURRENT_FRONTIER.md`.
2. Взять только один текущий фронт.
3. Сформулировать один структурный вопрос.
4. Сделать 1-3 точечные проверки.
5. Сразу записать, что эти проверки означают.
6. Если появился повторяющийся механизм, оформить мини-лемму.
7. Если механизм не появился, записать провал и сменить разрез.

## Три режима работы

### 1. Разведка

Используется, когда непонятно, какая клетка или строка важна.

Разрешено:

```text
короткая диагностика
rowscores / rowhist / branches
сравнение нескольких соседних веток
```

Запрещено:

```text
долго закрывать ветки без объяснения
```

Выход из режима разведки:

```text
найден кандидат на маркер или разрез
```

### 2. Гипотеза

Используется, когда появился возможный структурный механизм.

Нужно записать:

```text
какие клетки заданы
какая цепочка E677 их связывает
какой прогноз делает гипотеза
на каких соседних ветках ее надо проверить
```

Пример текущей гипотезы:

```text
u-transfer / no-tail trap
s=6*5
u=s*6
=> 5*u=4
=> 6*0=u
q=u*6
=> 6*(0*q)=0
```

### 3. Проверка

Используется, когда гипотеза уже сформулирована.

Проверки должны быть точечными:

```text
фиксированный s
фиксированный u
фиксированный q
или маленький заранее объясненный набор таких узлов
```

Если проверка закрыла узел, сразу записать:

```text
это подтверждает гипотезу
это противоречит гипотезе
это только локальное закрытие без общего смысла
```

## Правило против закапывания

После 3-5 похожих закрытий агент обязан остановиться на короткую внутреннюю
проверку:

```text
мы получили новый структурный факт?
или просто закрыли еще несколько случаев?
```

Если нового факта нет:

```text
1. записать это в research_log.md;
2. не повышать лимит автоматически;
3. сменить разрез или формулировку гипотезы.
```

## Таймауты

Таймаут сам по себе не является плохим результатом.

Каждый таймаут нужно классифицировать:

```text
широкий домен
плохой разрез
не хватает следствия E677
нужен следующий релейный маркер
нужна ролевая формулировка вместо чисел
```

Если один и тот же точечный узел дважды таймаутит и не дает новой структуры,
нужно записать это и сменить подход.

## Экономия запусков инструментов

Перед серией вычислительных проверок агент должен один раз убедиться, какой
runtime реально работает в текущем окне.

Дополнение 2026-05-25: `tools/search_counterexample_strong.js` поддерживает
узкий диагностический запрет пары в extra requirements:

```text
a:t:5~5:6
```

Смысл:

```text
not (a*t=5 and a*5=6)
```

Этот синтаксис добавлен только для проверки no-bridge веток в
`marker_bridge_transfer_lemma.md`. Не использовать его как замену полноценному
доказательству; он нужен для короткой диагностики отрицательных условий.

Если системный `node` указывает на WindowsApps и дает `Access is denied`, не
повторять такие запуски и не тратить контекст на дополнительные проверки
системного `node`. Сразу перейти к рабочему пути:

```text
Node REPL / встроенный runtime Codex
```

Правило:

```text
1. один короткий runtime-check;
2. если node.exe не запускается, больше не пробовать тот же путь;
3. использовать уже рабочий Node REPL wrapper для tools/search_counterexample_strong.js;
4. в журнал записывать только математический результат, а не техническую
   возню с runtime, если она не влияет на воспроизводимость.
```

Цель: не расходовать лимиты на повторяющиеся технические сбои, когда
математический протокол проверки уже понятен.

## Режим экономии лимитов и разделение ролей

Добавлено 2026-05-24.

По умолчанию дальнейшая работа должна идти в режиме экономии токенов и
лимитов:

```text
главный агент = стратегия, математика, выбор разреза, решение что считать;
субагенты = узкие проверки, сверка файлов, короткие вычислительные задачи.
```

Главный агент отвечает за:

```text
1. формулировку структурного вопроса;
2. выбор текущего фронта;
3. решение, нужна ли проверка вообще;
4. математическую интерпретацию результата;
5. обновление CONTINUE_HERE.md и research_log.md.
```

Субагента можно запускать только тогда, когда задача уже сведена к короткой
проверке. Хороший формат задачи для субагента:

```text
проверь u=8 для s=1,3,4 по уже заданному q-протоколу;
сверь, какие ветки s закрыты в файлах;
выпиши только domain для 6*0 и q;
проверь, повторяется ли уже сформулированная схема.
```

Плохой формат задачи:

```text
поищи лемму вообще;
подумай шире;
продолжай исследование;
реши case45 самостоятельно.
```

Ограничение на результат субагента:

```text
5-10 строк вывода;
только факты, статусы, домены, таймауты и короткий вывод;
без длинного рассуждения и без нового широкого плана.
```

Если доступный инструмент позволяет выбирать уровень рассуждения, главный
агент должен оставлять самый высокий уровень для стратегии, а субагентов
запускать на достаточном, но более дешевом уровне для узких проверок. Если
инструмент не позволяет явно выбрать уровень, это правило все равно действует
на уровне роли: субагент не принимает стратегических решений.

Практическое правило выбора уровня для субагентов:

Дополнение 2026-06-03: если главный агент запущен на высоком уровне, но не на
очень высоком, субагенты по умолчанию должны быть `gpt-5.5 medium`. В этом
режиме главный агент обязан уменьшать задачи до сложности medium: сверка
статуса, проверка одной уже заданной цепочки, 1-3 фиксированных узла, короткая
таблица доменов или таймаутов. `gpt-5.5 high` для субагентов использовать
только как исключение, когда узкая проверка реально слишком тонкая для medium и
без нее блокируется стратегическое решение.

Дополнение 2026-06-04: считать этот режим текущим режимом по умолчанию, если
пользователь явно не сказал, что главный запуск идет на очень высоком уровне.
То есть главный агент работает как стратег на `high`, а субагенты получают
только задачи, подходящие для `gpt-5.5 medium`.

Практическая граница:

```text
можно делегировать:
  сверить статус 1-3 уже названных узлов;
  проверить одну конкретную цепочку E677;
  выписать компактную таблицу доменов/таймаутов;
  найти в логах подтверждение уже сформулированного статуса.

нельзя делегировать medium-субагенту:
  найти главную лемму;
  выбрать следующий широкий разрез;
  решить, стала ли схема настоящей леммой;
  вести открытую разведку без заранее заданного протокола.
```

Если задача для субагента выглядит шире этого списка, главный агент обязан
сначала сузить ее до конкретной проверки или выполнить стратегическую часть
самостоятельно.

```text
gpt-5.5, средний уровень:
  сверка статуса математических веток;
  проверка, можно ли считать ветку закрытой;
  выписать покрытие представителей без самостоятельного обобщения;
  подготовить короткую таблицу "проверено / не проверено".

gpt-5.5, высокий уровень:
  не использовать по умолчанию, когда главный агент сам запущен на high;
  только исключительная узкая математическая проверка по уже заданному протоколу;
  сравнить 2-4 похожих узла, если medium уже недостаточен;
  проверить конкретную релейную цепочку, если ошибка здесь меняет статус леммы;
  найти противоречие в фиксированном узком остатке, если это блокирует стратегию.

gpt-5.5, низкий уровень:
  только нематематическая механика:
  сверка файлов;
  выписать домены/таймауты из логов;
  найти нужные строки;
  проверить наличие файлов;
  подготовить черновую таблицу без вывода "закрыто / не закрыто".

очень высокий уровень:
  не использовать для субагентов по умолчанию;
  резервировать для главного агента, который держит стратегию и лемму.
```

Важное усиление: в этой задаче статус "ветка закрыта" является математическим
выводом. Поэтому такие задачи нельзя отдавать слабому субагенту. Если нужно
решить, закрыт ли фронт, достаточно ли представителей, или можно ли поднять
кандидат до леммы, использовать не ниже `gpt-5.5` со средним или высоким
уровнем.

Для этого исследования не использовать `gpt-5.4-mini` по умолчанию. Даже
механическую работу лучше отдавать `gpt-5.5` на низком уровне, чтобы сохранить
единый стиль понимания проекта и снизить риск неверной интерпретации статуса.

Субагента на очень высоком уровне запускать только если пользователь явно
попросил или если есть короткая, но действительно трудная математическая
проверка, от которой зависит выбор всей стратегии. В обычном случае главный
агент сам делает этот стратегический вывод.

Грязная работа по умолчанию не должна выполняться главным агентом, если ее
можно безопасно изолировать. Под грязной работой здесь понимается:

```text
пройти несколько однотипных q-узлов;
сверить длинные журналы;
собрать таблицу статусов;
выписать домены;
перепроверить, какие случаи уже закрыты;
подготовить короткий список таймаутов;
проверить несколько представителей по уже заданной схеме.
```

Такая работа должна уходить:

```text
1. сначала в локальный инструмент/скрипт, если это чистое вычисление;
2. затем к субагенту среднего или высокого уровня, если нужна аккуратная
   сверка или маленькая математическая проверка;
3. главному агенту - только итог и математическая интерпретация.
```

Главный агент может выполнить грязную работу сам только если:

```text
задача занимает один-два коротких шага;
нет доступного субагента;
стоимость запуска субагента выше самой проверки;
результат нужен немедленно для выбора следующего разреза.
```

После ответа субагента главный агент обязан:

```text
1. принять или отклонить результат;
2. записать математический смысл в основной журнал;
3. не копировать длинный отчет целиком, если достаточно короткого итога.
```

## Субагенты

Субагентов запускать только для узких задач.

Хорошая задача для субагента:

```text
проверь по файлам, какие s-ветки уже закрыты
проверь узлы s=2,3 по протоколу u,q
сформулируй ролевой вывод из уже сохраненных данных
```

Плохая задача для субагента:

```text
поищи лемму вообще
подумай над задачей широко
продолжай исследование
```

Результат субагента должен быть интегрирован в главный журнал или явно
отклонен.

## Формат сохранения результата

После каждого значимого этапа обновлять:

```text
CONTINUE_HERE.md
research_log.md
соответствующий файл леммы
```

Если изменилась карта файлов, обновить:

```text
RESULTS_INDEX.md
PROJECT_STATUS.md
```

## Текущий стандарт продолжения

Последний закрытый фронт:

```text
7*0=4
6*6=6
status: closed
```

Текущий стандарт продолжения:

```text
1. не возвращаться внутрь закрытого узла 6*6=6;
2. восстановить по журналу статус соседних узлов 6*6=7 и 6*6=8;
3. если они открыты, сформулировать ролевой аналог u-transfer / no-tail trap;
4. сделать только 1-3 короткие проверки, чтобы понять переносится ли механизм;
5. если механизм не переносится, записать границу леммы и сменить разрез.
```
