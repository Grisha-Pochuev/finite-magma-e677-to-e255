# Self-Swap Lemma Candidate

Дата: 2026-05-21.

## Зачем эта записка

Релейная схема закрыла локальный хвост `case 45`, но не стала общей леммой для
соседних веток `7*0=2,3,5`. Поэтому следующий более крупный кандидат - не
сама релейная цепочка, а маленький узор внутри активной строки.

## Обозначения

В `case 45` нулевая строка является полным 9-циклом:

```text
0*0=1, 0*1=2, ..., 0*8=0
```

Удобная обратная нотация:

```text
b1=8, b2=7, b3=6, b4=5, b5=4, b6=3, b7=2, b8=1
```

В оставшихся верхних ветках пишем:

```text
t = b2*0
```

Лестница дает:

```text
b3*t=b4
```

Для `case 45` это:

```text
7*0=t
6*t=5
```

## Новый кандидат

Закрытая ветка `7*0=4, 6*6=6, 6*5=4` имеет особое свойство:

```text
t=b5
b3*t=b4
b3*b3=b3
b3*b4=t
```

То есть строка `b3`:

```text
фиксирует b3
и меняет местами b4 и b5
```

В числах:

```text
6*6=6
6*5=4
6*4=5
```

Это не просто `s=t`. Для `t=2` и `t=3` тоже можно поставить `s=t`, но тогда
строка `6` меняет местами `5` и дальний элемент `t`, а не соседнюю пару
`b4,b5`. Именно поэтому соседние ветки остаются намного шире.

## Релейное ядро self-swap

Из E677 и обратной цепочки в этом узоре получается:

```text
u = b5*b3
b4*u=b5

v = b4*b5
v*b4=b3
```

Это левое релейное плечо. В числах:

```text
u = 4*6
5*u=4

v = 5*4
v*5=6
```

Есть и симметричное правое плечо:

```text
w = b4*b3
b5*w=b4

r = b5*b4
r*b5=b3
```

В числах:

```text
w = 5*6
4*w=5

r = 4*5
r*4=6
```

Практически сильным оказалось левое плечо. Именно по нему были закрыты все
значения `4*6`:

```text
4*6=0 -> противоречие
4*6=1 -> закрыто
4*6=2 -> закрыто
4*6=3 -> закрыто
4*6=4 -> противоречие
4*6=5 -> противоречие
4*6=7 -> закрыто
4*6=8 -> закрыто
```

В циклических ролях это выглядит так:

```text
u=4*6=0   -> запрещенный плохой край b4*0=b5
u=4=b5   -> противоречие
u=5=b4   -> противоречие
u=3=b6   -> включается известный следующий край b4*b6=b5
u=8=b1   -> реле переносит в старшую строку
u=7=b2   -> реле переносит в старшую строку
u=2=b7   -> релейный хвост закрывается коротким активным разрезом
u=1=b8   -> релейный хвост закрывается коротким активным разрезом
```

Это полезнее, чем список "закрыто": оно показывает, какие роли должен
покрывать будущий ручной довод.

## Каскадная интерпретация

Особенно важен вариант:

```text
u=b6
```

Тогда левое плечо `b4*u=b5` превращается в:

```text
b4*b6=b5
```

Это уже не произвольный подслучай, а сдвиг той же логики на шаг ниже по
циклу:

```text
b3*b5=b4
b3*b4=b5

переходит к ребру

b4*b6=b5
```

В числах:

```text
6*4=5
6*5=4

при 4*6=3 получается

5*3=4
```

Именно поэтому дальше естественно возникла клетка `5*4`: реле продолжает
каскад вдоль соседних элементов цикла, а не выбирает новую клетку случайно.

Рабочая картина теперь такая:

```text
self-swap запускает каскад соседних ребер;
u=0 упирается в запрещенный плохой край;
u=b6 сдвигает каскад вниз по циклу;
u=b1,b2,b7,b8 пытается уйти через старшие/дальние роли;
в меньших циклах такой уход может попасть во внешний tail;
в полном 9-цикле tail отсутствует, и каскад остается запертым внутри цикла.
```

Это пока не доказательство, но это уже лемма-кандидат более высокого уровня:

```text
в полном цикле соседний self-swap порождает замкнутый каскад релейных ребер,
который не имеет внешнего выхода и должен упереться в запрещенный плохой край.
```

### Каскадный узел `u=b6`

Ветка `u=b6` в `case 45`:

```text
4*6=3
5*3=4
```

в циклических ролях:

```text
b5*b3=b6
b4*b6=b5
```

Диагностика показала, что строка `b4` резко сжимается. Это не случайность:
две клетки выводятся структурно.

1. Из `b5*b3=b6=L_0^{-1}(b5)` обратная лестница по нулевой строке дает:

```text
b4*0=b3
```

В числах:

```text
5*0=6
```

2. Затем E677 при `x=0` дает:

```text
если y*0=a и a*y=b, то y*(0*b)=0
```

Для `y=b4`, `a=b3`, `b=b5`:

```text
b4*b4=0
```

В числах:

```text
5*5=0
```

Именно это было видно в домене строки `5`:

```text
5*0=6 forced
5*3=4 forced
5*5=0 forced
```

Дальше рассматривается следующий соседний обмен:

```text
v=b4*b5=b6
```

В числах:

```text
5*4=3
```

Тогда строка `5=b4` имеет всего 12 вариантов, а клетка `5*6=b4*b3`
имеет только два значения:

```text
5*6=2=b7
5*6=8=b1
```

Проверка:

```text
5*6=8 -> contradiction
5*6=2 -> remains, but forces a very small residue
```

При оставшемся варианте `5*6=2` строка `5` получает почти полный цикл:

```text
5*0=6
5*6=2
5*2=5
5*5=0
5*3=4
5*4=3
```

В ролях:

```text
b4*0=b3
b4*b3=b7
b4*b7=b4
b4*b4=0
b4*b6=b5
b4*b5=b6
```

После этого строка `6=b3` имеет только 10 вариантов. Лучший последний разрез:

```text
6*8
```

Результат:

```text
6*8=0 -> contradiction
6*8=1 -> 4 variants, all dead
6*8=2 -> contradiction
6*8=7 -> contradiction
6*8=8 -> contradiction
```

Значит каскадный узел:

```text
u=b6, v=b6
```

закрывается коротким деревом:

```text
5*6 in {b7,b1}
b1 branch contradictory
b7 branch -> split by b3*b1 -> all dead
```

Это важный шаг к настоящей лемме: один из раньше "вычислительных" хвостов
стал маленьким структурным узлом с понятными ролями.

### Полное дерево узла `u=b6`

Для узла:

```text
u=b6
```

то есть:

```text
4*6=3
5*3=4
```

возможные значения следующего релейного шага:

```text
v=5*4 in {1,2,3,7,8}
```

Ролевая карта:

```text
v=2=b7 -> contradiction
v=7=b2 -> contradiction
v=1=b8 -> row b4 has 10 variants, all dead
v=8=b1 -> row b4 has 6 variants, 5 dead; last residue dies on row b3
v=3=b6 -> shifted cascade, closed by the subnode above
```

В числах для `v=8=b1` последний остаток:

```text
row 5 = 6 3 5 4 8 0 2 7 1
row 6 domain = 6
all 6 row-6 variants dead
```

Следовательно весь узел:

```text
u=b6
```

закрыт коротким ролевым деревом, а не длинным перебором.

Правое плечо пока выглядит слабее как первый разрез: в уже проверенной
диагностике `5*6` оставалось широкой клеткой, тогда как `4*6=0` стало
структурно запрещенным и запустило закрывающий маршрут.

## Почему `t=4` отличается от `t=2,3`

Короткая диагностика одинакового self-подтипа:

```text
7*0=t
6*t=5
6*6=6
6*5=t
```

дала такие размеры строк:

```text
t=2: row 2 domain 74322
t=3: row 3 domain 66478
t=4: row 4 domain 51131
t=5: row 5 domain 5760
```

Но главное отличие не только в размере домена. Для `t=4=b5` клетка среднего
шага `4*6` сразу становится заметным разрезом:

```text
t=4: 4*6 has value 0 only 417 times inside row 4 domain
```

Для `t=2,3` соответствующие клетки `2*6` и `3*6` остаются широкими:

```text
t=2: 2*6 minimum bucket 6539
t=3: 3*6 minimum bucket 5754
```

Значит сжатие в `t=4` связано не с общей формой `s=t`, а с соседним обменом
`b4 <-> b5` внутри строки `b3`.

## Запрещенный нулевой край

Для плохого элемента `0` уже известно общее запрещение:

```text
y*0 != L_0^{-1}(y)
```

Если в self-swap ядре `u=b5*b3=0`, то из `b4*u=b5` получается:

```text
b4*0=b5=L_0^{-1}(b4)
```

Это запрещено. Поэтому ветка:

```text
4*6=0
```

закрывается не вычислительно, а структурно.

## Проверка переносимости

Проверен более простой вариант гипотезы: "если есть такой же соседний
self-swap в меньших циклах, то он должен немедленно закрываться".

Тесты:

```text
case 44: b2=6, b3=5, b4=4, b5=3
6*0=3, 5*3=4, 5*5=5, 5*4=3

case 42: b2=5, b3=4, b4=3, b5=2
5*0=2, 4*2=3, 4*4=4, 4*3=2

case 39: b2=4, b3=3, b4=2, b5=1
4*0=1, 3*1=2, 3*3=3, 3*2=1
```

Результат короткой диагностики:

```text
case 44: first row b3, domain 422, first 30 candidates: 22 dead
case 42: first row b3, domain 499, first 30 candidates: 27 dead
case 39: first row b3, domain 398, first 30 candidates: 30 dead
```

Вывод: self-swap не является мгновенной доказанной леммой, но ведет себя как
сильный структурный узор за пределами `case 45`. Особенно заметно, что в
меньших циклах первой активной строкой снова становится `b3`.

Дополнительная проверка `case 39` более широким начальным куском уточнила
картину:

```text
case 39: checked 120 of 398 first-row candidates
dead after first row: 63
live samples remain, next rows include b4, b5 and row 1
```

Значит первые 30 мертвых вариантов были сильным, но недостаточным сигналом.
Self-swap пока доказывает не универсальное закрытие, а устойчивую активность
строки `b3`. Для общей леммы нужно дополнительное условие: либо полный 9-цикл,
либо еще один инвариант, который отличает закрытый `case 45` от живых
продолжений в меньших циклах.

## Текущий статус кандидата

Не доказанная лемма, но сильный кандидат:

```text
Если плохой элемент имеет длинный цикл, t=b5, b3*t=b4,
b3*b3=b3 и b3*b4=b5, то self-swap ядро должно вести к противоречию.
```

Более осторожная формулировка, уже подтвержденная разведкой:

```text
Соседний self-swap b3 fixes b3 and swaps b4,b5 создает релейное ядро,
в котором нулевой средний шаг запрещен структурно, а остальные шаги
резко сжимаются через строки b4,b5,b3 и старшие строки.
```

## Следующий осмысленный шаг

Не возвращаться к длинному перебору `case 45`. Следующая математическая цель:

```text
доказать или опровергнуть self-swap lemma в меньших циклах case 39/42/44,
начиная не с полного поиска, а с релейного ядра:

u=b5*b3
b4*u=b5
v=b4*b5
v*b4=b3
p=v*b3
p*v=b5
```

Если это ядро закрывается одинаково для `case 39/42/44/45`, тогда получится
настоящая кандидат-лемма, а не локальный маршрут одной ветки.

После расширенной проверки `case 39` это надо читать осторожнее: ядро не
закрывается одинаково сразу. Поэтому следующий вопрос стал точнее:

```text
что именно добавляет полный 9-цикл case 45 к self-swap узору?
```

## Остановка 2026-05-21: что осталось после узла `u=b6`

Каскадный узел:

```text
u=4*6=3=b6
```

оформлен хорошо и закрыт как маленькое дерево ролей. После этого коротко
сравнены оставшиеся значения среднего шага:

```text
u=1=b8
u=2=b7
u=7=b2
u=8=b1
```

Эти проверки нужны были не для нового перебора, а чтобы понять, являются ли
они такими же каскадными узлами, как `u=b6`.

Результат:

```text
u=1=b8:
  row 5 domain 6487
  branches: first row 6, domain 78
  checked 50, dead 31
  следующий живой слой часто идет в row 1

u=2=b7:
  row 6 domain 108

  живые продолжения уходят в row 7, row 2, row 8

u=7=b2:

  живые продолжения уходят в row 8, row 7, row 1
```

Вывод: оставшиеся значения `u` пока не дают такого компактного каскадного
узла, как `u=b6`. Они выглядят как перенос в старшие или дальние строки
`b1,b2,b7,b8` и требуют отдельного общего принципа. Старое реле уже закрывало
эти ветки, но для ручной леммы еще нужно объяснить этот перенос.

Точка продолжения:

```text
не повторять проверку u=b6;
она уже оформлена.

следующий вопрос:
почему u in {b1,b2,b7,b8} переносит работу в строки 1,7,8,2
и как это выразить одной леммой о старшем/дальнем переносе?
```

## Новый инвариант после проверки: tail escape

Сравнение с меньшими циклами показало вероятную причину, почему self-swap
не закрывается там мгновенно. В `case 39/42/44` нулевая строка не является
полным 9-циклом:

```text
case 39: zero-cycle [0,1,2,3,4,5], tail [6,7,8]
case 42: zero-cycle [0,1,2,3,4,5,6], tail [7,8]
case 44: zero-cycle [0,1,2,3,4,5,6,7], tail [8]
case 45: zero-cycle [0,1,2,3,4,5,6,7,8], tail empty
```

Живые образцы self-swap в меньших циклах используют этот внешний хвост:

```text
case 39, first 80 checked:
dead after first row 38/80
live samples have tail values in column 4 in 20/20 printed samples

case 42, first 60 checked:
dead after first row 40/60
live samples have tail values in columns 5,6,7,8

case 44, first 60 checked:
dead after first row 33/60
live samples have tail value 8 mainly in senior columns 6,7
```

Интерпретация: self-swap сам по себе не закрывает все циклические типы. Он
создает давление на активную строку `b3`, а меньшие циклы могут сбрасывать
это давление во внешний хвост нулевой строки. В полном `case 45` такого хвоста
нет, поэтому тот же узор превращается в закрывающий механизм.

Контрольная проверка самого `case 45` важна: отсутствие хвоста не убивает
первую строку мгновенно.

```text
case 45 self-swap, first row b3=6
first domain 483
checked 80
dead after first row 45
live samples remain
```

Значит `no tail escape` - не самостоятельное одномоментное противоречие. Более
точно: живые варианты в полном 9-цикле не могут уйти во внешний хвост и поэтому
остаются внутри одного релейного ядра; именно это ядро затем закрыло все
значения `4*6`.

Новая рабочая формулировка:

```text
self-swap + no tail escape
```

То есть настоящая лемма может быть не просто про соседний обмен `b4 <-> b5`,
а про соседний обмен внутри полного цикла левой трансляции `L_0`.
