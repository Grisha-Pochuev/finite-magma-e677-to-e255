# Классификация веток по значению r2

Дата: 2026-05-19

Эта записка расширяет аудит `case45_active_row_audit.md`. Цель - смотреть не
только на чистый 9-цикл `case 45`, а на всю длинную зону размера `9`, cases
`34-45`.

## Общая схема

Пусть `0` - плохой элемент, а

```text
b_j = L_0^{-j}(0)
r_j = b_j*0
```

Главная лестница:

```text
b_j * r_{j-1} = b_{j+1}
```

Для текущей зоны важен первый настоящий параметр плохости:

```text
r_2 = b_2*0
```

Если зафиксировано

```text
r_2=t
```

то лестница сразу дает:

```text
b_3*t = b_4
```

Значит после выбора `r_2` естественно смотреть не произвольные клетки, а
строку `b_3`, особенно:

```text
b_3*0
b_3*b_3
```

Это обобщает наблюдение из `case 45`, где `b_2=7`, `b_3=6`, `b_4=5`, и
поэтому при `7*0=t` получалось `6*t=5`.

## Проверка на cases 34-45

Для оставшихся длинных циклов размера `9` значения `b_1,b_2,b_3,b_4` такие:

```text
cases 34-38, цикл длины 5: b1=4, b2=3, b3=2, b4=1
cases 39-41, цикл длины 6: b1=5, b2=4, b3=3, b4=2
cases 42-43, цикл длины 7: b1=6, b2=5, b3=4, b4=3
case 44, цикл длины 8: b1=7, b2=6, b3=5, b4=4
case 45, цикл длины 9: b1=8, b2=7, b3=6, b4=5
```

Диагностика соседних cases подтвердила общий принцип.

### Case 44

Здесь:

```text
b2=6, b3=5, b4=4
```

При обычном значении:

```text
6*0=2
```

лестница дает:

```text
5*2=4
```

Диагностика выбирает прежде всего:

```text
5*5
5*0
```

то есть строку `b_3=5`.

### Case 42

Здесь:

```text
b2=5, b3=4, b4=3
```

При:

```text
5*0=2
```

лестница дает:

```text
4*2=3
```

Диагностика выбирает:

```text
4*0
4*4
```

то есть снова строку `b_3=4`.

## Особые значения r2

Общий принцип `b_3*t=b_4` остается верным всегда, но разные значения `t`
создают разные типы веток.

### Обычный тип

Если `t` не является одним из специальных соседей `b_1`, `b_2`, `b_4`, то
главная активная строка - `b_3`.

Рабочий следующий разрез:

```text
b_3*b_3
```

и рядом:

```text
b_3*0
```

Так как в контрпримере `r_2` не может быть равно `b_3`, имеем `t != b_3`.
Строка `b_3` является перестановкой, поэтому из двух фактов

```text
b_3*t=b_4
t != b_3
```

сразу следует маленькая лемма:

```text
b_3*b_3 != b_4
```

Это объясняет, почему в `case 45` при `b_3=6`, `b_4=5` значение `6*6=5`
вообще не появляется.

### Само-тип `r2=b2`

Это ветка:

```text
b_2*0=b_2
```

Она уже изучалась раньше. В размере `9` закрыт общий подслучай:

```text
b_2*0=b_2
b_2*b_2=b_{m-1}
```

Но весь само-тип как общая лемма еще не завершен.

### Перенос назад `r2=b1`

Если:

```text
r_2=b_1
```

то диагностика показывает включение строки `b_1`.

Примеры:

```text
case 45: 7*0=8=b1 -> активна строка 8
case 39: 4*0=5=b1 -> активна строка 5
```

Это объясняет, почему ветка `7*0=8` в `case 45` была особой и закрывалась
маршрутом через `8*0`, а не через обычное заполнение строки `6=b_3`.

### Перенос вперед `r2=b4`

Если:

```text
r_2=b_4
```

то из общей формулы получается:

```text
b_3*b_4=b_4
```

То есть строка `b_3` фиксирует элемент `b_4`. После этого начинает включаться
строка `b_4`.

Примеры:

```text
case 45: 7*0=5=b4 -> после 6*5=5 заметна строка 5
case 44: 6*0=4=b4 -> после 5*4=4 заметна строка 4
```

## Вывод

Теперь главный объект исследования шире, чем один хвост `case 45`.

Рабочая классификация:

```text
r2=b1  -> особый перенос к строке b1
r2=b2  -> само-тип b2
r2=b4  -> перенос к строке b4
обычные r2 -> активная строка b3
```

Запрещенные значения в контрпримере:

```text
r2 != 0
r2 != b3
```

Следующий осмысленный шаг: для обычного типа попытаться сформулировать
лемму о строке `b_3`:

```text
если b_3*t=b_4 и t не является b1,b2,b4, то строка b_3 вынуждает противоречие
после малого числа параметров b_3*b_3 и b_3*0.
```

Если эта лемма не выходит, надо отдельно разбирать три особых типа:

```text
r2=b1
r2=b2
r2=b4
```

Пользовательский ноутбук сейчас не нужен.

## Второй уровень: self-тип на строке b3

После обычного выбора `r_2=t` и перехода к строке `b_3` следующий важный
параметр:

```text
q = b_3*b_3
```

Для любого значения `q` есть контрольное соотношение из E677. Так как
`b_3*b_3=q`, то `L_{b_3}(b_3)=q`, а значит `L_{b_3}^{-1}(q)=b_3`.
По формуле

```text
L_y^{-1}(x)=x*((y*x)*y)
```

получаем:

```text
q * ((b_3*q)*b_3) = b_3
```

Практический смысл: если `q != b_3`, то клетка `b_3*q` должна быть сильным
следующим маркером, потому что она входит в выражение, которое возвращает
`b_3`.

Если же

```text
q=b_3
```

это соотношение вырождается в тавтологию и почти не давит на таблицу. Поэтому
self-тип второго уровня структурно труднее остальных значений `q`.

В self-типе нужно использовать не `q`, а другой известный прообраз. Так как

```text
b_3*t=b_4
```

то `L_{b_3}^{-1}(b_4)=t`. Снова применяем

```text
L_y^{-1}(x)=x*((y*x)*y)
```

при `y=b_3`, `x=b_4`:

```text
t = b_4 * ((b_3*b_4)*b_3)
```

Значит в self-типе естественный не вырожденный маркер:

```text
s = b_3*b_4
```

После этого условие становится:

```text
b_4*(s*b_3)=t
```

То есть структурный выход из self-типа состоит из трех шагов:

```text
1. s = b_3*b_4
2. u = s*b_3
3. b_4*u = t
```

Это объясняет, почему простой выбор самых узких клеток строки `b_3` может
увести в глубокое дерево: настоящий выход из self-типа проходит через
связку строк `b_3 -> s -> b_4`.

Пробная диагностика в текущем хвосте `case 45`

```text
7*0=4, 6*6=6
```

при выборе

```text
s=6*5=7
```

показала, что строка `5=b_4` сужается, но не становится маленькой сама по
себе. Это подтверждает уточнение: нужно учитывать средний шаг `u=s*b_3`, а не
переходить к строке `b_4` напрямую.

Текущий трудный хвост `case 45`

```text
7*0=4, 6*6=6
```

имеет смысл не как случайная ветка, а как self-тип второго уровня:

```text
b_3*b_3=b_3
```

Проверка для обычных значений `t=2,3,4` в `case 45` показала, что при

```text
7*0=t
6*6=6
```

диагностика все равно остается внутри строки `6=b_3`:

```text
t=2: важны 6*8, 6*1, 6*7, 6*4, 6*5
t=3: важны 6*1, 6*8, 6*2, 6*7, 6*5
t=4: важны 6*1, 6*2, 6*7, 6*8, 6*0
```

Это объясняет, почему хвост `6*7=2/3` тяжелый: мы попали не в произвольный
нижний случай, а в повторение само-типа на следующей строке.

Новая формулировка продолжения:

```text
1. Первый уровень: выбрать r2=b_2*0=t.
2. Обычный тип: перейти в строку b_3, потому что b_3*t=b_4.
3. Второй уровень: классифицировать q=b_3*b_3.
4. Если q=b_3, это self-тип второго уровня, и его надо изучать как отдельный
   структурный случай, а не как хвост конкретной ветки.
```

Техническое усиление инструмента: в `tools/search_counterexample_strong.js`
добавлен режим `rowscores`, который считает `cellscores` только для выбранных
строк. Это нужно, чтобы проверять роли `b_1`, `b_3`, `b_4` без перегрузки
памяти.

## Соседние специальные значения q

После self-типа были точечно проверены значения

```text
q=b_2
q=b_1
```

в `case 44` при обычном `r2=6*0=2`.

### `q=b_2`

```text
5*5=6=b_2
```

Строка `5=b_3` резко сжимается: домен строки около `1187`, первые маркеры:

```text
5*0, 5*3, 5*1, 5*7, 5*8, 5*4, 5*6
```

### `q=b_1`

```text
5*5=7=b_1
```

Домен строки больше, около `2547`, первые маркеры:

```text
5*6, 5*1, 5*4, 5*7, 5*3, 5*8, 5*0
```

Вывод: `q=b_2` и `q=b_1` тоже остаются задачами строки `b_3`, но не
вырождаются как self-тип `q=b_3`. Для них контрольное соотношение

```text
q * ((b_3*q)*b_3) = b_3
```

должно давать реальное давление на таблицу.

## Разветвление по s=b3*b4 в текущем self-типе

В текущем хвосте `case 45`:

```text
t=4
b3=6
b4=5
q=6*6=6
s=6*5
```

Проверены возможные роли `s`. Диагностика показывает:

```text
s=1 -> активна строка 1, особенно 1*1
s=2 -> строка 2 остается очень широкой
s=3 -> строка 3 остается широкой, но важны 3*6 и 3*0
s=4=t -> строка 4 активна, и главный маркер 4*6 совпадает со средним шагом u=s*b3
s=7=b2 -> активна строка 7, особенно 7*7
s=8=b1 -> активна строка 8, особенно 8*0 и 8*8
```

Это дает следующую классификацию третьего уровня:

```text
s=b1 или s=b2 -> перенос к уже важным старшим строкам
s=t          -> средний шаг u=s*b3 становится главным маркером
обычные s    -> строка s может оставаться широкой
```

Для ветки `s=t=4` проверен средний шаг:

```text
u = 4*6
```

Результат:

```text
u=0 -> противоречие
u=4 -> противоречие
u=5 -> противоречие
u=3 -> строка 5=b4 сжимается до маленького домена
u=1,2,7,8 -> остаются живыми средними хвостами
```

Противоречие при `u=0` имеет понятное объяснение в текущем случае:

```text
если u=0, то b4*0=t=b5
```

а для плохого элемента запрещено `r_4=b_4*0=b_5`.

Попытка напрямую закрыть маленький хвост `u=3` коротким поиском дала таймаут
на этапе проверки доменов, то есть его не надо добивать грубой силой.

## Проверка self-типа второго уровня вне case 45

Чтобы не подгонять вывод под чистый 9-цикл, self-тип

```text
b_3*b_3=b_3
```

проверен в соседних cases.

### Case 44

```text
b2=6, b3=5, b4=4
r2=6*0=2
q=5*5=5=b3
```

Режим `rowscores` по строке `5` показал, что первые маркеры остаются в строке
`5`:

```text
5*7, 5*1, 5*6, 5*4, 5*3, 5*0
```

### Case 42

```text
b2=5, b3=4, b4=3
r2=5*0=2
q=4*4=4=b3
```

Режим `rowscores` по строке `4` показал:

```text
4*5, 4*7, 4*8, 4*0, 4*6, 4*3
```

Вывод: self-тип второго уровня повторяется за пределами `case 45`. Поэтому
его надо изучать как общий структурный случай:

```text
r2 обычного типа
b_3*r2=b_4
q=b_3*b_3=b_3
```

а не как частный хвост `6*6=6`.
