# Senior/Far Transfer Lemma Candidate

Дата: 2026-05-21.

## Контекст

Эта записка продолжает `self_swap_lemma.md`.

Рабочая зона `case 45`:

```text
b1=8, b2=7, b3=6, b4=5, b5=4, b6=3, b7=2, b8=1
```

Self-swap:

```text
b3*b3=b3
b3*b4=b5
b3*b5=b4
```

Левое релейное плечо:

```text
u=b5*b3
b4*u=b5
```

Узел `u=b6` уже оформлен как каскад вниз по циклу. Остались значения:

```text
u in F = {b1,b2,b7,b8}
```

В числах:

```text
u in {8,7,2,1}
```

## Основная идея

Дальний `u` не является новым независимым случаем. Он передает давление в
следующий релейный узел:

```text
v=b4*b5
v*b4=b3

p=v*b3
p*v=b5
```

В числах:

```text
v=5*4
v*5=6

p=v*6
p*v=4
```

То есть работа переносится из клетки `u=b5*b3` в строку `v`, а строка `b3`
остается главным перевалочным пунктом.

## Почему активируется строка `b3`

Ключевой ручной шаг: строка `b3` становится активной не эмпирически, а
принудительно.

В полном цикле:

```text
0*b4=b3
L_0^{-1}(b4)=b5
```

Обратная bad-column лестница говорит:

```text
если строка x имеет L_0^{-1}(x) в единственной колонке c,
то (0*x)*0=c.
```

Берем:

```text
x=b4
0*x=b3
L_0^{-1}(x)=b5
```

А строка `b4` уже имеет:

```text
b4*u=b5
```

Значит:

```text
b3*0=u
```

В числах:

```text
5*u=4  ==>  6*0=u
```

Это объясняет все диагностические строки вида:

```text
u=1 -> row 6 col 0 = 1
u=2 -> row 6 col 0 = 2
u=7 -> row 6 col 0 = 7
u=8 -> row 6 col 0 = 8
```

Именно поэтому первый реальный collapse почти всегда идет через строку `b3`.
Дальний `u` становится не внешним случаем, а значением плохой колонки строки
`b3`.

## Нулевой маркер внутри строки `b3`

После:

```text
b3*0=u
```

следующий естественный параметр:

```text
q=u*b3
```

E677 при `x=0` дает общее правило:

```text
если y*0=a и a*y=q,
то y*(0*q)=0
```

Для `y=b3`, `a=u`:

```text
q=u*b3  ==>  b3*(0*q)=0
```

Это объясняет форму живых строк `b3`: они в первую очередь различаются тем,
в какой колонке появляется ноль. В полном 9-цикле колонка `0*q` всегда
остается внутри того же цикла, то есть ноль не может уйти во внешний tail.

Таким образом senior/far transfer имеет два связанных маркера:

```text
bad-column marker: b3*0=u
zero-position marker: q=u*b3 -> b3*(0*q)=0
```

Это может быть ключом к ручному доказательству active-row collapse второго
уровня.

Representative-проверка для ветки:

```text
u=b1, v=b8, p=b7
```

дала:

```text
q=u*b3 in {0,b8,b7}
```

В числах:

```text
q=8*6 in {0,1,2}
```

По правилу `q -> b3*(0*q)=0`:

```text
q=0 -> 6*1=0
q=1 -> 6*2=0
q=2 -> 6*3=0
```

Split по `q`:

```text
q=0 -> row 6 domain 6, all dead
q=1 -> row 6 domain 24, 22/24 dead; residues row 2 domain 9 and row 1 domain 12
q=2 -> row 6 domain 22, 15/22 dead; residues row 1 domain 12/13
```

Это объясняет двухступенчатый collapse: сначала `q` ставит ноль в строке
`b3`, затем оставшиеся маленькие far-row residues умирают.

Вторая representative-пара подтверждает тот же механизм.

Для:

```text
u=b7, v=b2, p=b8
```

в числах:

```text
4*6=2
5*2=4
5*4=7
7*5=6
7*6=1
1*7=4
```

получено:

```text
q=u*b3=2*6 in {0,b8,b7,b2}
```

то есть:

```text
q=0 -> 6*1=0
q=1 -> 6*2=0
q=2 -> 6*3=0
q=7 -> 6*8=0
```

Split:

```text
q=0 -> contradiction
q=1 -> row 2 domain 13, all dead
q=2 -> row 6 domain 18, 17/18 dead; one residue row 2 domain 23
q=7 -> row 6 domain 16, all dead
```

Значит `q`-маркер повторяется: он ставит ноль в строке `b3`, после чего
остаются только маленькие far-row residues.

## Общая q-карта для senior/far пар

Проверены все пары:

```text
u in F={b1,b2,b7,b8}
v in F\{u}
```

Для каждой пары смотрелся только домен:

```text
q=u*b3
```

Результат в числах:

```text
u=1=b8:
  v=2 -> q in {1,2}
  v=7 -> q in {0,1,2}
  v=8 -> q in {0,1,2}

u=2=b7:
  v=1 -> q in {0,1,2,7}
  v=7 -> q in {0,1,2,7}
  v=8 -> q in {0,1,2,7}

u=7=b2:
  v=1 -> q in {0,1,2,7}
  v=2 -> q in {0,1,2,7}
  v=8 -> q in {0,1,2,7}

u=8=b1:
  v=1 -> q in {0,1,2}
  v=2 -> q in {0,1,2}
  v=7 -> contradiction
```

Ролевая форма:

```text
q=u*b3 всегда лежит в {0,b8,b7,b2}
```

Иными словами, senior/far transfer запрещает:

```text
q in {b1,b3,b4,b5,b6}
```

Это очень сильное сжатие. Вместе с правилом:

```text
q -> b3*(0*q)=0
```

оно означает, что ноль в строке `b3` может стоять только в колонках:

```text
0*q in {b8,b7,b6,b1}
```

В числах:

```text
q=0 -> 6*1=0
q=1 -> 6*2=0
q=2 -> 6*3=0
q=7 -> 6*8=0
```

Новая подлемма-кандидат:

```text
В senior/far transfer параметр q=u*b3 не может попасть в среднюю полосу
{b1,b3,b4,b5,b6}; он зажат в {0,b8,b7,b2}.
```

Этот факт теперь почти полностью выведен вручную.

Из q-маркера:

```text
q=u*b3  ==>  b3*(0*q)=0
```

исключаются:

```text
q=b1:
  0*q=0, значит b3*0=0,
  но b3*0=u != 0.

q=b4:
  0*q=b3, значит b3*b3=0,
  но b3*b3=b3.

q=b5:
  0*q=b4, значит b3*b4=0,
  но b3*b4=b5.

q=b6:
  0*q=b5, значит b3*b5=0,
  но b3*b5=b4.
```

Остается тонкое значение `q=b3`. Оно запрещается collision-следствием E677.

Если:

```text
q=u*b3=b3
```

то две разные строки `u` и `b3` дают одинаковое значение в колонке `b3`:

```text
u*b3=b3
b3*b3=b3
```

Collision-следствие:

```text
если y*x=z*x=c, y != z,
то (y*c)*y=(z*c)*z
```

дает:

```text
(u*b3)*u=(b3*b3)*b3
```

то есть:

```text
b3*u=b3
```

Но строка `b3` уже имеет:

```text
b3*b3=b3
```

и `u != b3`. Это нарушает перестановочность строки `b3`.

Итого доказана подлемма:

```text
q=u*b3 in {0,b8,b7,b2}
```

для senior/far transfer.

## Размер остатка после q-подлеммы

После выбора `q` строка `b3` имеет принудительные клетки:

```text
b3*0=u
b3*b3=b3
b3*b4=b5
b3*b5=b4
b3*(0*q)=0
```

Это пять разных колонок и пять разных значений. Поэтому одна только
перестановочность оставляет:

```text
4! = 24
```

варианта строки `b3`.

Это совпадает с диагностикой:

```text
q=1 -> row 6 domain 24
q=2 -> row 6 domain 22 или 18
q=7 -> row 6 domain 16
q=0 -> часто contradiction или domain 6
```

Следовательно senior/far transfer уже сведен к маленькой строковой лемме:

```text
после q-подлеммы остается не большой case 45,
а максимум 24 варианта строки b3.
```

Это допустимый уровень для отдельной ручной/solver lemma, но следующий шаг
должен быть именно доказательством этой 24-вариантной строки, а не возвратом
к большому дереву случаев.

## Доказательство релейного переноса

Из известного ребра:

```text
b4*u=b5
```

по обратной цепочке:

```text
a*z=c  ==>  z=c*((a*c)*a)
```

получаем:

```text
u = b5*((b4*b5)*b4)
```

Но в этом плече уже известно:

```text
u=b5*b3
```

Так как строка `b5` является перестановкой:

```text
(b4*b5)*b4=b3
```

Обозначим:

```text
v=b4*b5
```

Тогда:

```text
v*b4=b3
```

Теперь используем известное self-swap ребро:

```text
b3*b5=b4
```

Применяем релейную форму к:

```text
v*b4=b3
b3*b5=b4
```

и получаем:

```text
(v*b3)*v=b5
```

Обозначим:

```text
p=v*b3
```

Тогда:

```text
p*v=b5
```

## Четыре типа выхода по `v`

Вместо четырех отдельных `u` надо классифицировать `v=b4*b5`.

### 1. Zero-return

```text
v=0
```

Тогда:

```text
b4*b5=0
0*b6=b5
```

Реле дает:

```text
h=b4*0
h*b4=b6
```

В числах:

```text
h=5*0
h*5=3
```

### 2. Diagonal

```text
v=u
```

Тогда:

```text
p*u=b5
b5*b3=u
```

Еще один релейный шаг дает:

```text
r=p*b5
r*p=b3
```

В числах:

```text
p=u*6
p*u=4
r=p*4
r*p=6
```

### 3. Middle Return

```text
v=b6
```

Это возвращает работу к уже оформленному каскадному узлу:

```text
u=b6
```

или к shifted-варианту, где возникают клетки:

```text
5*0=6
5*5=0
5*6 in {2,8}
6*8 closes the residue
```

### 4. Senior/Far Transfer

```text
v in F \ {u}
```

Тогда реле переносит работу в строку `v`:

```text
p=v*b3
p*v=b5
```

Диагностически живые остатки уходят в строки:

```text
row 1, row 2, row 7, row 8
```

но снова возвращаются через активную строку `b3`.

## Candidate Lemma

```text
В полном L0-цикле соседний self-swap

b3*b3=b3
b3*b4=b5
b3*b5=b4

и off-cascade значение

u=b5*b3 in {b1,b2,b7,b8}

не создают нового независимого хвоста. Они вынуждают релейный перенос:

v=b4*b5
v*b4=b3
p=v*b3
p*v=b5

После этого каждый выход по v является одним из четырех типов:
zero-return, diagonal, middle return, senior/far transfer.
```

Что пока не доказано полностью:

```text
нужно чисто алгебраически закрыть senior/far transfer,
то есть показать, что v in F\{u} неизбежно возвращается
в diagonal, zero-return или каскадный узел.
```

```

p=4,q=1 -> none
p=4,q=2 -> none
p=8,q=1 -> none
p=8,q=2 -> none
```

Трудный узел

```text
u=b8, v=b7, p=b6, q=b7
```

имеет строку `b3` с 16 вариантами. Все 16 вариантов закрылись короткой
проверкой. Для одного большого остатка

```text
row b3 = 1 8 3 0 5 4 6 7 2
```

найден одно-клеточный убийца:

```text
b8*b5 = 1*4
```

и все его возможные значения противоречивы:

```text
1*4=0 -> contradiction
1*4=4 -> contradiction
1*4=8 -> contradiction
```

Для остальных вариантов строки `b3` также видны маленькие одно-клеточные
закрытия, чаще через клетки:

```text
2*0
1*4
3*1
3*0
3*4
1*2
```

Это уточняет последнюю форму senior/far-кандидата:

```text
senior/far transfer
-> q-подлемма
-> совместный (p,q)-слой
-> строка b3 имеет малый домен
-> каждый остаток закрывается одной маленькой far-row клеткой
```

Еще одна новая диагностическая карта: для всех проверенных пар с `u=2` и `u=7`
параметр `q` остается в уже доказанной четверке `{0,b8,b7,b2}`, а `p` остается
маленьким:

```text
u=2, v=1: p in {1,2,3,4},   q in {0,1,2,7}
u=2, v=7: p in {1,2,3,8},   q in {0,1,2,7}
u=2, v=8: p in {1,2,3,4,8}, q in {0,1,2,7}

u=7, v=1: p in {1,2,3,4},   q in {0,1,2,7}
u=7, v=2: p in {1,2,3,4,8}, q in {0,1,2,7}
u=7, v=8: p in {1,2,3,4,8}, q in {0,1,2,7}
```

Продолжать нужно не с повторной проверки `u=b8,v=b7`, а с поиска ручного
объяснения последнего одно-клеточного закрытия. Самый конкретный следующий
объект:

```text
u=b8, v=b7, p=b6, q=b7
row b3 = 1 8 3 0 5 4 6 7 2
доказать вручную, почему клетка 1*4 не может быть 0, 4 или 8
```

Если это получится, следующий шаг - обобщить эту "far-row killer cell" схему на
все 16 вариантов строки `b3`, а затем проверить оставшиеся senior/far пары с
`u=8` и `u=1`.
