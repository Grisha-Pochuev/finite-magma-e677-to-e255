# Row 6 Compact Residual Lemma

Дата: 2026-05-23.

Эта записка фиксирует новый объект после закрытия локальной леммы
`u-transfer / no-tail trap` для узла:

```text
case45
7*0=4
6*6=6
```

## Откуда берется объект

Ветка `7*0=4` всегда дает лестничное ребро:

```text
6*4=5
```

Поэтому для любого значения `r=6*6`, не только для self-типа `r=6`, работает
та же обратная цепочка:

```text
s=6*5
u=s*6
5*u=4
6*0=u
q=u*6
6*(0*q)=0
```

Это важное расширение: `u-transfer` не зависит от условия `6*6=6`. Оно
использует только ребро `6*4=5` и полный цикл строки `0`.

## Проверка на соседних узлах

Соседние узлы `6*6=7` и `6*6=8` действительно снова активируют строку `6`.

Для

```text
7*0=4
6*6=7
```

диагностика строки `6` дала домен `878`. Лучшие клетки снова лежат в строке
`6`, включая:

```text
6*2
6*0
6*1
6*8
6*3
```

Для

```text
7*0=4
6*6=8
```

диагностика строки `6` дала домен `1554`. Лучшие клетки снова лежат в строке
`6`, включая:

```text
6*7
6*1
6*2
6*8
6*3
6*5
6*0
```

## Первый перенос `u-transfer`

Проверены два зеркальных тестовых узла.

### Узел A

```text
7*0=4
6*6=7
6*5=8
```

После фиксации `6*5=8` домен строки `6` сжался с `878` до `218`.
Клетка `6*0` стала одной из главных:

```text
6*0 in {1,2,3,4,6}
```

Это согласуется с прогнозом:

```text
u=8*6=6*0
```

### Узел B

```text
7*0=4
6*6=8
6*5=7
```

После фиксации `6*5=7` домен строки `6` сжался с `1554` до `224`.
Клетка `6*0` стала особенно узкой:

```text
6*0 in {1,2,3}
```

Это снова согласуется с прогнозом:

```text
u=7*6=6*0
```

## Компактный остаток

Проверен один конкретный переносимый узел:

```text
7*0=4
6*6=8
6*5=7
u=7*6=6*0=1
```

Тогда

```text
q=1*6
6*(0*q)=0
```

Диагностика дала:

```text
q in {0,1,2,6}
```

Короткая проверка закрыла:

```text
q=0 -> closed
q=6 -> closed
```

Но два значения дали одинаковый тяжелый остаток:

```text
q=1 -> timeout, но строка 6 имеет только 13 вариантов
q=2 -> timeout, но строка 6 имеет только 16 вариантов
```

Это не широкий провал. Это новый компактный объект.

Для `q=1` строка `6` имеет вид:

```text
6*0=1
6*2=0
6*4=5
6*5=7
6*6=8
```

Остальные значения распределяются по четырем свободным колонкам. Всего
остается `13` вариантов строки `6`.

Для `q=2` строка `6` имеет вид:

```text
6*0=1
6*3=0
6*4=5
6*5=7
6*6=8
```

Остальные значения также распределяются по четырем свободным колонкам. Всего
остается `16` вариантов строки `6`.

## Текущий вывод

`u-transfer / no-tail trap` переносится за пределы self-узла `6*6=6`, но в
соседней ветке он не закрывает все сразу. Он приводит к новому объекту:

```text
компактный остаток строки 6
```

Кандидат на следующую лемму:

```text
Если в ветке 7*0=4 строка 6 содержит блоки

6*0=1
6*k=0
6*4=5
6*5=7
6*6=8

где k=2 или k=3, то такой компактный остаток несовместим с E677.
```

Это пока не доказано. Простое дробление по `6*7` не помогло: все четыре
значения `6*7 in {2,3,4,6}` для хвоста `q=1` дали одинаковый таймаут на
первом узле. Значит следующий шаг - не повышать лимит, а искать новый
структурный маркер для этого компактного остатка.

## Обновление: хвост `q=1` закрыт полным релейным шагом



### `v=4`


### `v=6`

```text
В компактном остатке q=1 нельзя дробить только 6*7.
Нужно брать полный релейный шаг:

6*7=v
v*6=h
7*h=5

После этого строка 6 сжимается до 2-4 вариантов и закрывается.
```

## Обновленная мини-лемма компактного остатка

В компактном остатке после

```text
6*6=8
6*5=7
7*6=1
6*0=1
```

значения `q=1*6` закрываются так:

```text
q=0,6 -> closed directly
q=1,2 -> full relay through 6*7=v, v*6=h, 7*h=5
```

Если после этого строка `6` фиксируется полностью, появляется еще один
обратный перенос в строку `3`:

```text
6*a=3
6*3=0
=> 3*7=a
```

для `a=1` или `a=2`.

## Обновление: `u=2, q=0` закрыт; `u=2, q=2` выявил вторичный self-type

Для

```text
u=2
q=0
```

строка `6` имела домен `7`. Полный релейный шаг из `6*5=7` дал:

```text
6*7=3 -> contradiction
6*7=4 -> contradiction
6*7=6 -> require 7*8=5
```

После `7*8=5` остаток закрылся по `6*2`:

```text
6*2=1 -> closed
6*2=3 -> closed
6*2=4 -> closed
```

Итог:

```text
u=2, q=0 -> closed
```

Для

```text
u=2
q=2
```

получается вторичный self-type:

```text
2*6=2
6*3=0
```

Это объясняет, почему хвост тяжелее: активной после частичного закрытия
становится уже строка `2`, а не строка `6`.

Частично проверено:

```text
6*7=3 -> closed by full relay
6*7=6 -> mostly closed by full relay, but one fixed row-6 residual remains
```

Открытый фиксированный остаток внутри `u=2,q=2,6*7=6`:

```text
row 6 = 2 4 1 0 5 7 8 6 3
```

В нем уже задано:

```text
2*6=2
```

и диагностика выбирает строку `2`. Это новый уровень проблемы, а не просто
еще один вариант строки `6`.

Новая гипотеза:

```text
трудные компактные остатки возникают, когда q=u, то есть u*6=u.
Это вторичный self-type. Его надо разбирать как self-узел строки u,
а не продолжать дробить строку 6.
```

## Остановка: начат разрез вторичного self-type

Для вторичного self-type

```text
u=2
q=2
2*6=2
```

начат более естественный разрез. После аудита 2026-05-24 оставлена только
доказанная часть:

```text
s=6*2
h=s*6
2*h=0
```

Смысл: это прямое обратное следствие известного ребра `6*0=2`.

Ранее здесь была добавлена связь:

```text
3*0=h
```

Она не следует из указанной обратной цепочки. Поэтому старое объяснение
проверки `s=6 -> h=8 -> closed` было неверным.

После аудита `s=6` был перепроверен без условия `3*0=h` и все равно закрылся:

```text
s=6 -> status none, около 12 секунд
```

Значит факт закрытия `s=6` сохраняется, но его нужно объяснять через валидную
цепочку `2*h=0`, а не через `3*0=h`.

Историческая точка перед закрытием:

```text
u=2
q=2
2*6=2
s=6*2
h=s*6
2*h=0
checked: s=6 closed
then-open: s in {1,3,4}
```

Главный вывод не менялся: `q=u` является вторичным self-type, и продолжать
только дробить строку `6` неправильно. Эта историческая точка закрыта ниже
через валидный релейный выход из строки `2`.

```text
status after update below: closed
```

Исторически оставалось после `u=2`:

```text
u=3
```

## Обновление 2026-05-24: cross-pair закрыт

Широкий поворот из этой записки завершился отдельной локальной леммой в
`row6_cross_pair_transfer_lemma.md`.

Закрыты обе ориентации:

```text
6*6=8, 6*5=7 -> closed
6*6=7, 6*5=8 -> closed
```

Для зеркальной стороны:

```text
u=8*6=6*0
q=u*6
6*(0*q)=0
```

дало полное закрытие:

```text
u=6 -> closed
u=1 -> closed
u=2 -> closed
u=3 -> closed
u=4 -> closed
```

Значит compact residual был не тупиком, а частью более широкой cross-pair
схемы. Следующий объект - низкие значения `6*5`, а не старые хвосты внутри
`6*5=7`.
